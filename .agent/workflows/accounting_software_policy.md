# 会計ソフト別対応方針

## 対応順序

1. **MF（マネーフォワード）を最優先で完成させる**
2. freee・弥生はMF完了後に着手

## 理由

- MFはMCP API連携があり、科目・税区分の同期が自動化できる
- freee・弥生はAPI連携なし。CSV出力→手動インポートの運用
- freee・弥生にMF形式CSVをインポートする場合、CSV列構造の変換が必要（そのままでは取り込めない）
- 科目名・税区分名の不一致時は会計ソフト側で手動マッピングが必要
- この変換・マッピング対応は工数が大きく、MF完成前に着手する意味がない

## 現状の出力

- CSVエクスポートはMF形式のみ（exportMfCsv.ts）
- 弥生・freee用のCSV列定義は定義済み（accountingConstants.ts SOFTWARE_EXPORT_CSV_SCHEMAS）
- 弥生・freee用の税区分変換マッピング定義も定義済み（accountingConstants.ts SOFTWARE_TAX_MAPPINGS_TEXT）
- ただし上記定義を使うCSV出力コードは未実装（TaxCodeMapper.ts, CsvExportService.tsは削除済み）

## freee・弥生対応時にやること（MF完了後）

- 弥生インポート形式のCSV出力関数を実装
- freeeインポート形式のCSV出力関数を実装
- 税区分名の変換（MF名→弥生名/freee名）を実装
- 顧問先のaccountingSoftwareに応じて出力形式を切り替え

## 全社マスタ・MCPインポートの設計方針

### 1. 全社勘定科目マスタの意義

- スグスルは税理士事務所向けの業務アプリであり、複数の顧問先（法人/個人）を一元管理する
- 全社マスタは全顧問先共通のテンプレート。MF連携・未連携に関わらず、全顧問先にデフォルトの勘定科目と税区分を提供する
- MF未連携の顧問先もスグスルを利用するため、全社マスタが必要

### 2. 全社税区分マスタの意義

- 税区分は課税方式（原則一括比例/原則個別対応/簡易/免税）で利用可否が異なる
- 全社マスタで151件の税区分を管理し、課税方式ごとの利用可否（mf-tax-available.json）で表示を制御する

### 3. MCPインポートの目的

- MF（マネーフォワード）のMCP APIから科目・税区分のデフォルト値を取得し、全社マスタに反映する
- 手動設定では不正確になるため（D3で24件の誤りを検出した実績あり）、MCP実機データを正とする
- MFがデフォルト科目を追加・廃止した場合に、全社マスタに自動反映する（削除候補検知）

### 4. デフォルト法人/個人アカウント（TST/TSK）の目的

#### 前提条件

- MFのAPIには「デフォルト科目一覧」「デフォルト税区分一覧」を返す専用エンドポイントが存在しない
- `getAccounts`/`getTaxes`が返すのは常に「その事業者の現在の状態」であり、ユーザーがMF管理画面で変更した後の値が返る
- ユーザーはMF管理画面で科目のデフォルト税区分を変更できる（例: 「旅費交通費」の税区分を「課税仕入10%」→「対象外」に変更）
- 全社マスタは全顧問先共通のテンプレートであり、特定1社のカスタマイズを反映してはならない

#### 専用アカウントの設計

- TST（c_rODnkCDN / 株式会社すぐする）: 法人のMFデフォルトアカウント。科目133件
- TSK（c_wTdnMKDO / XXXX）: 個人のMFデフォルトアカウント。科目108件
- MF側で科目・税区分を一切カスタマイズしない（デフォルト状態を維持）
- 運用中の顧問先からインポートすると、その顧問先固有のカスタマイズ（税区分変更等）が全社マスタに混入し、他の全顧問先に波及する。これを防ぐために専用アカウントを用意している
- 税区分はTST1社で4課税方式を順番に変更してインポートすることで、4方式全てのデータを取得済み

### 5. 名前照合の理由

- MFのID（科目ID・税区分ID）は事業者固有。TSK vs TSTで0/151件一致（MCP実機検証済み）
- そのため全社マスタにはMF IDを保存しない。名前照合でマスタIDに変換して保存する

### 6. スグスルID（税区分ID・勘定科目ID）の役割

- MF生IDは事業者固有で全社マスタに保存できない → 会計ソフト非依存の中間キーが必要
- スグスルの税区分ID（例: `PURCHASE_TAXABLE_10`）はMF税区分名（例:「課税仕入 10%」）と1:1対応。名前照合で決定論的に紐づく
- スグスルIDが存在する理由: 将来freee・弥生に広げる際、会計ソフトごとの税区分名の差をオーバーライドテーブル1枚で吸収するため（taxCategoryMappings.ts）
- MFだけなら名称で十分だが、マルチ会計ソフト対応のためにスグスルIDを介在させる

### 7. インポート時のMF生ID媒介

- MFの科目レスポンスには税区分名が入っておらず、`tax_id`（MF生ID）しかない
- 税区分名を得るにはMF税区分一覧で`tax_id`→名前を引く必要がある（MFのAPI仕様の制約）
- 変換チェーン: MF科目の`tax_id` → MF税区分名 → スグスル税区分マスタと名前照合 → スグスルの`taxCategoryId`
- MF生IDは変換パイプラインの途中で使うだけで捨てる。保存されるのはスグスルの税区分IDのみ
- 科目の照合自体は`mf.name`（科目名）で全社マスタと名前照合。MF科目IDは使わない

### 8. インポート元の顧問先を変更しても崩壊しない理由

- MFデフォルト科目の名前は全事業者共通（法人133件固定、個人108件固定。MCP実機検証11パターンで差分ゼロ）
- MF税区分の名前も全事業者共通（151/151件一致）
- 名前照合でスグスルIDに変換するので、どの顧問先から取得しても結果は同一
- ただしユーザーがMF管理画面で科目のデフォルト税区分を変更した場合、その顧問先からインポートすると全社マスタが変更後の値で上書きされる → TST/TSKはデフォルト状態を維持する運用（§4）で回避

### 9. 課税方式による`tax_id`（MF生ID）の変動

- 免税 vs 原則: 差分0件（法人・個人とも）
- 原則一括比例 vs 原則個別対応: 差分0件（法人・個人とも）
- 簡易 vs 原則個別対応: 法人4件・個人7件が変動（売上系のみ）
- 変動内容: 「課税売上 10%」→「課税売上 10% 三種」のように簡易課税の事業種別が付加されるだけ。方向（sales/purchase/common）は変わらない

### 10. `taxDetermination`（税区分判定ルール）の問題

- `auto_purchase`（自動仕入）/ `auto_sales`（自動売上）/ `fixed`（固定）の3値分類
- STREAMEDの概念を模倣したもの。MFには存在しない概念
- `deriveTaxDetermination()`は`accountGroup`（5値）だけで一律判定 → 粗すぎて53件の矛盾が発生
- **矛盾の本質:** MFの151種の税区分を3値に無理やり潰す設計自体が情報欠落を起こす。同じ`PL_EXPENSE`（費用）でも「旅費交通費」は課税仕入、「給料賃金」は対象外。この違いを`accountGroup`だけでは表現できない
- `defaultTaxCategoryId`（既定税区分ID）の`direction`（方向）で直接判定すれば、3値への中間変換は不要
- バリデーション#9（科目×税区分不整合）も`defaultTaxCategoryId`の`direction`で再構成可能

### 11. 設計原則（2026-06-09確立）

1. **MFが正。** MFサーバーが返した実機データが唯一の根拠
2. **MF生IDには依存しない。** 名称ベースで照合し、スグスルの中間IDを介在させる
3. **全社マスタにMF生IDを保存しない。** 事業者固有のIDを全社テンプレートに持つのは設計矛盾
4. **151種→3値の中間分類は情報欠落。** 科目ごとの`defaultTaxCategoryId`の`direction`で直接判定すべき
5. **インポート元を変えても結果は同一。** MFデフォルトの名前が全事業者共通だから
6. **TST/TSKはMFデフォルト状態を維持する。** カスタマイズした顧問先からのインポートで全社マスタが汚染されることを防ぐ
   7.MF側にマイナスは送信できず、マイナス仕訳は使えないので振替方式のみ
