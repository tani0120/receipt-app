### 小さく開発への方針転換

**背景**:
- Phase 3-B（UIモック29件）は複雑すぎて実装困難
- 仕訳UI + 顧問先登録UIに絞る必要性

**決定内容**:
1. **L4/L5を「凍結」**（段階的に硬化）
2. **MVP Scope**: 仕訳UI + 顧問先登録UI
3. **Layer A/B/C組み合わせ**:
   - Layer A: 段階的確認（技術的エラー排除）
   - Layer B: Mapping Table（L4代替）
   - Layer C: Service単体テスト + 型での禁止（L5代替）

**L4/L5の本質**:
- L4 = 「意味と画面を1対1に束縛する層」
- L5 = 「意味が壊れていないことを機械が保証する層」

**過去の痛み**:
- 「表示されない」: どのUIが責任を持って表示するか未定義
- 「デバッグ不能」: 設計違反か実装ミスか判別不能
