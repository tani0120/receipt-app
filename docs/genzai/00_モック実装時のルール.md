## 2. ディレクトリ安全構造

以下のフォルダはいかなる理由があっても**修正・新規追加しない**。

```
src/database/types/   ← DB型定義
src/shared/           ← 共有型定義
src/stores/           ← Pinia Store（※例外あり。下記参照）
```

理由: 型安全性の基礎。汚染されると全体が連鎖的に崩壊する。

> **「新規追加」も禁止（2026-03-04追加）**
> 既存ファイルの変更だけでなく、新しいファイルの作成も禁止。
> 「既存を壊していない」は免罪符にならない。レイヤー汚染は破壊と同等に扱う。

#### shared に置いてよいもの / 置いてはいけないもの

| ✅ 置いてよい | ❌ 置いてはいけない |
|---|---|
| 純粋型定義（interface / type） | 外部サービス仕様（MF CSV定義等） |
| 純粋enum | UI表示定数（null表示文字列等） |
| 値オブジェクト | モック専用ロジック |
| インフラ非依存ユーティリティ | 実装方針メモ・設計ドキュメント |

> 判断基準: 「このファイルを削除したとき、Domain型の整合性に影響があるか？」
> Noなら shared に置く必然性がない。`src/mocks/definitions/` 等に配置せよ。

#### shared 変更の例外プロセス（2026-03-04追加）

sharedへの追加が真に必要な場合は、以下を**全て**経ること:

1. **フェーズ移行宣言**（Phase B以降であること）
2. **影響範囲の明示**（どの層が影響を受けるか）
3. **人間のレビュー承認**

> アーキテクチャ固定フェーズでは「追加 = 構造変更」と同義。
> AIの自己判断による追加は禁止。

#### stores/ の例外（2026-03-03追加）

以下の条件を全て満たす場合のみ、stores/ の変更を許可する:

1. **画面固有の設定ストア**である（例: `settingsStore.ts` — 基本情報パネルの値を画面間共有）
2. **Domain型のビジネスロジック**を含まない（計算・バリデーション・API呼び出し等は禁止）
3. **ref + return のみ**の単純な状態管理である
4. 変更内容がフィールド追加のみ（既存フィールドの型変更・削除は別途承認が必要）

> 判断基準: 「このストアを丸ごと削除してもビジネスロジックが壊れない」なら画面固有ストア。

```
src/mocks/
  unsafe/        ← any許可（隔離・import禁止）
  data/          ← Domain型必須（any禁止）
  components/    ← Domain型必須（any禁止）
  types/         ← 聖域（any絶対禁止）
  definitions/   ← UI/外部連携仕様の定義定数（2026-03-04追加。sharedから移動）
```

### ESLint ルール（物理的制約）
- `unsafe/` から `data/` `components/` へのimport → **error**
- `unsafe/` 以外で `any` 使用 → **error**
- `as any` → **error**
- `@ts-ignore` → 理由コメント+TODO必須

### 3.4 列タイプ別の実装パターン

#### Phase A-0: 実装済みパターン（getValue + type分岐）

```typescript
// getValue: データパスkey（debit.account等）に対応するユーティリティ
function getValue(obj: any, path: string): unknown {
  return path.split('.').reduce((o: any, k: string) => o?.[k], obj)
}
```

```vue
<template v-for="col in journalColumns" :key="col.key">
  <!-- text型: journal-level（keyにドットなし）/ entry-level（keyにドットあり） -->
  <template v-if="col.type === 'text'">
    <div v-if="!col.key.includes('.') && rowIndex === 0" :class="col.width">
      {{ getValue(journal, col.key) }}
    </div>
    <div v-else-if="col.key.includes('.')" :class="col.width">
      {{ getValue(row, col.key) }}
    </div>
    <div v-else :class="col.width"></div>
  </template>

  <!-- amount型: undefined/null対策付き -->
  <template v-else-if="col.type === 'amount'">
    <div :class="col.width">
      {{ getValue(row, col.key) != null ? Number(getValue(row, col.key)).toLocaleString() : '' }}
    </div>
  </template>

  <!-- component型: col.key個別分岐（10列） -->
  <template v-else-if="col.type === 'component'">
    <template v-if="col.key === 'photo'">
      <i class="fa-solid fa-camera" ...></i>
    </template>
    <template v-else-if="col.key === 'needAction'">
      <!-- 4FAアイコン: NEED_DOCUMENT / NEED_INFO / REMINDER / NEED_CONSULT -->
      <div class="flex gap-0.5">
        <button v-for="noteKey in staffNoteKeys" :key="noteKey"
                @click="toggleStaffNote(journal.id, noteKey)" ...>
          <i :class="staffNoteConfig[noteKey].icon"></i>
        </button>
      </div>
    </template>
    <!-- 他のcomponent型列も同様にcol.key個別分岐 -->
  </template>
</template>
```

**Phase A-0の成果**:
- 全列のwidthが `journalColumns.ts` に集約される
- 物理的にwidth不一致が防げる
- `getValue()`でデータパスkey（`debit.account`等）に対応
- `key.includes('.')`でjournal-level/entry-level判定（Vue側ロジック）

#### Phase B: getCellComponent方式（綺麗に整理）

```typescript
import type { Component } from 'vue'

const cellComponents: Record<string, Component> = {
  photo: PhotoCell,
  pastJournal: PastJournalCell,
  comment: CommentCell,
  needAction: NeedActionCell,
  labelType: LabelTypeCell,
  warning: WarningCell,
  rule: RuleCell,
  taxRate: TaxRateCell,
  memo: MemoCell,
  invoice: InvoiceCell,
  // 列追加時はここに1行追加するだけ
}

function getCellComponent(key: string): Component | undefined {
  return cellComponents[key]
}
```

```vue
<template v-for="col in journalColumns" :key="col.key">
  <!-- text/amount型: getValue()経由 -->
  <template v-if="col.type === 'text'">
    <div :class="col.width">{{ getValue(row, col.key) }}</div>
  </template>
  <template v-else-if="col.type === 'amount'">
    <div :class="col.width">{{ getValue(row, col.key)?.toLocaleString() }}</div>
  </template>

  <!-- component型: 動的コンポーネント -->
  <template v-else-if="col.type === 'component'">
    <component :is="getCellComponent(col.key)" :journal="row" />
  </template>
</template>
```

**Phase Bでの整理**:
- v-if連鎖を解消
- セルコンポーネントに分離
- 列追加時は `cellComponents` に1行追加するだけ
- テンプレートは一切変更不要

### 3.5 列定義設計原則

- columns.tsは構造定義のみ。描画ロジックはVue側。
- keyは原則データパス（例: `transaction_date`, `debit.sub_account`）。component/checkbox/index/action型はUI識別子。
- 配列順 = UI表示順。
- type: 8種（checkbox/index/component/text/amount/action/account-dropdown/category-dropdown）。account-dropdown/category-dropdownは2026-03-19 D7/D7aで追加。
- sortKeyのみcolumns.tsで定義。ソート処理はVueコンポーネント側。

### 将来的な自動化（Phase B以降）

**pre-commit hook** でスコープ検証を自動化する:

```javascript
// .githooks/scope-check.js（Node.js）
const fs = require('fs');
const { execSync } = require('child_process');

// スコープ宣言ファイル読み込み
if (!fs.existsSync('.scope-declaration.txt')) {
  console.error('❌ エラー: スコープ宣言がありません');
  process.exit(1);
}

const declared = fs.readFileSync('.scope-declaration.txt', 'utf8')
  .split('\n').filter(Boolean);
const changed = execSync('git diff --cached --name-only')
  .toString().split('\n').filter(Boolean);

const undeclared = changed.filter(f => !declared.includes(f));

if (undeclared.length > 0) {
  console.error('❌ スコープ外ファイルが変更されています:');
  undeclared.forEach(f => console.error(`  - ${f}`));
  process.exit(1);
}

console.log('✅ スコープ検証: 合格');
```

**運用フロー:**
1. 作業開始時: LLMが `.scope-declaration.txt` 作成
2. 作業中: 自由に作業
3. commit時: pre-commit hookが自動検証 → スコープ外変更はcommit拒否

**限界:**
- `--no-verify` でhookを無視できる → **人間が `git diff --stat` で最終確認**
- `.scope-declaration.txt` を書き換えられる → **人間がdiffで検出**

> 完全ではないが、暴走の90%は検出できる。残り10%は人間の最終確認で補う。

## 13. エンコーディングルール（2026-03-12追加）

> **経緯**: `journal_test_fixture_30cases.ts`がShift-JIS（CP932）で保存されており、仕訳一覧画面（`/journal-list/:clientId`）で全データ行が文字化けした。
> Node.jsの`TextDecoder('shift-jis')`でUTF-8に変換して修復済み。

### 原則

- **ソースコード**（`.ts` `.vue` `.md` `.json` `.css` `.html`）は全て**UTF-8**
- **CSVエクスポート**（MFクラウド会計向け）は**UTF-8 BOM付き**（`\uFEFF`を先頭に付与）
- **CSV読み込み**（銀行・クレカ明細）はShift-JISの可能性があるため、`TextDecoder('shift-jis')`で明示的に変換

### 物理的強制

| 手段 | 対象 | 効果 |
|------|------|------|
| `.editorconfig`に`charset = utf-8` | 全エディタ | 保存時にUTF-8を強制 |
| `.vscode/settings.json`に`files.encoding: utf8` | VS Code | UTF-8を強制 |
| huskyのpre-commitフックにエンコーディングチェック | git | Shift-JISファイルのコミットを拒否 |

### CSVエクスポート仕様（MFクラウド会計準拠）

- MFクラウド会計はUTF-8とShift-JISの両方に対応
- **UTF-8（BOM付き）を採用**（特殊文字の文字化け回避のため）
- BOM = `\uFEFF`（3バイト: `0xEF 0xBB 0xBF`）をCSVテキストの先頭に付与
- ExcelがBOMなしUTF-8をShift-JISと誤判定するのを防ぐ

## 15. localStorage保存ルール（2026-03-15追加）

> **経緯**: N2/N7修正（保存ボタンcomposable同期統一）で発覚した問題。
> 1. ページの`saveChanges()`がlocalStorage直書きし、composableの`overrides` refに反映しない → リロード後にデータ不整合
> 2. `rows`キー（全行データ）と`overrides`キー（差分データ）の二重管理 → hiddenIdsが矛盾
> 3. AIエージェント（subagent）のブラウザはPlaywright別プロセスでlocalStorageが分離 → テスト結果とユーザー環境が一致しない

### 保存ルール

| ルール | 内容 |
|--------|------|
| **composable経由保存** | ページの`saveChanges()`はcomposableの`saveAll()`を呼ぶ。localStorage直書き禁止 |
| **overrides refの同時更新** | `saveAll()`内で`overrides.value = { ... }`を更新してからlocalStorageに保存 |
| **hiddenIdsの導出元** | ページのrows（`deprecated`/`effectiveTo`）から導出。composableの`isHidden()`に頼らない |
| **二重管理の回避** | `rows`キーと`overrides`キーの両方を保存する場合、同一の情報源から導出すること |

### 4ページの保存ボタン統一パターン（2026-03-15完了）

| ページ | 保存方法 |
|--------|---------|
| マスタ勘定科目 | `accountRows` → `rows`キー + rowsから導出した`overrides`キーに保存 |
| マスタ税区分 | `allTaxRows` → `rows`キー + rowsから導出した`overrides`キーに保存 |
| 顧問先勘定科目 | `composable.saveAll(accountRows, subAccounts)` 経由 |
| 顧問先税区分 | `composable.saveAll(allTaxRows)` 経由 |

### subagentブラウザの分離

| 事実 | 対応 |
|------|------|
| subagentはPlaywright別プロセスでlocalStorageがユーザーと分離 | subagentのテスト結果でユーザーの環境は検証できない |
| subagentがlocalStorageを変更してもユーザーには影響しない | ユーザーのデータ確認はユーザー自身に依頼する |
| コードの動作検証はsubagent内で完結可能 | 同じコードが動くため、ロジックの正しさは検証できる |

> **推奨**: コード変更の検証は`vue-tsc`+コードレビューが主。ブラウザ動作確認はユーザーに依頼する。

### PowerShellの`Set-Content`禁止（2026-03-16追加）

| ルール | 理由 |
|--------|------|
| **`Set-Content`でソースファイルを書き換えてはならない** | PowerShellの`Set-Content`はデフォルトでUTF-16（BOM付き）を使用するため、UTF-8必須の`.vue`/`.ts`/`.js`/`.css`等を破壊する |
| ファイル編集は必ず`replace_file_content`/`multi_replace_file_content`/`write_to_file`ツールを使用する | これらはエンコーディングを保持する |
| やむを得ずPowerShellで書き換える場合は`-Encoding UTF8`を明示する | ただし非推奨。ツールを使うべき |

> **経緯**: 2026-03-15に`MockExportPage.vue`が`Set-Content`でUTF-16化し、IDE上で100+エラーが発生。`git checkout`で復元が必要になった。
