> 作成: 2026-05-21
> 準拠: [load_context.md](../../.agent/workflows/load_context.md) Supabase移行前倒し原則

## 背景

現状: AIコマンド実行のたびにMCPから全仕訳・PL試算表をリアルタイム取得。
問題: 遅い（24秒）、MCP障害で全停止、同じデータを何度も取得、名寄せも毎回AI呼び出し。

方針転換: **MCPは月1回の同期パイプ。sugu-sru側に3期分キャッシュ。AIコマンドはローカルDBから0.1秒で実行。**

## 1. 保管コスト試算

### データ量（1社あたり）

| 項目 | 件数 | サイズ |
|---|---|---|
| 仕訳（1年分） | 1,200件 × 1.5KB | 1.8MB |
| 仕訳（3期分） | 3,600件 | 5.4MB |
| PL試算表・推移表 | 12レコード/年 × 3期 | 0.4MB |
| 口座/カード/取引先マスタ | 約60件 | 0.1MB |
| 名寄せキャッシュ | 約30グループ | 0.01MB |
| **1社合計** | | **約6.0MB** |

### Supabase Pro（$25/月 ≒ ¥3,750）で何社まで？

| 条件 | 計算 | 結果 |
|---|---|---|
| DB上限 | 8GB | — |
| 1社あたり | 6.0MB | — |
| **追加料金なし** | 8GB ÷ 6.0MB | **約1,330社** |
| 自動スケーリング | $0.125/GB追加 | 上限なし |

> **200社なら1.2GB。8GBの15%しか使わない。1,330社まで追加料金なし。**

### ワーストケース（仕訳多い業種）

| ケース | 仕訳数/年 | 1社サイズ | 200社 | Proで何社？ |
|---|---|---|---|---|
| 通常（個人事業） | 1,200件 | 6.0MB | 1.2GB | 1,330社 |
| 飲食店（レシート多い） | 5,000件 | 24MB | 4.8GB | 333社 |
| EC事業（大量取引） | 10,000件 | 48MB | 9.6GB | 166社（+$0.2/月） |

### 保持ポリシー

```
保持: 今期 + 1期前 + 2期前 = 3期分
削除: 3期前以前 → 月次バッチで自動DELETE
```

## 2. 毎月取得タイミング

### 毎月20日

| 顧問先タイプ | 入力完了時期 | 20日同期の効果 |
|---|---|---|
| 自計化（自分で入力） | 翌月5〜15日 | ✓ ほぼ確定 |
| 記帳代行（事務所が入力） | 翌月10〜20日 | ✓ ほぼ確定 |
| 入力が遅い | 翌月末以降 | △ 翌月の同期で差分を拾う |
| 銀行自動連携のみ | 翌月1〜5日自動 | ✓ 完全に確定 |

> **20日でほぼ全社カバー。遅い社は翌月の同期で差分を拾う。**

```
基本: 毎月20日 00:00 JST
土日祝 → 翌営業日に自動繰越
```

## 3. 未確定仕訳の扱い

月次決算をしていない顧問先が多い。仕訳が未確定（`is_realized = false`）の場合:

### 3層で管理

```
DB層:   仕訳テーブルに is_realized カラム（MFの値そのまま保持）
集計層: 未確定含めて合算（除外しない。実態を反映）
表示層: 取引先名に ⚠ マーク（未確定仕訳を含む場合）
```

### レポート表示例

```
売上ランキング:
  | 順位 | 取引先 | 売上合計 | 構成比 |
  | 1 | A商事 | ¥1,000,000 | 49% |
  | 2 | B物産 ⚠ | ¥800,000 | 39% |
  | 3 | C工業 | ¥500,000 | 12% |

⚠ = この取引先の金額に未確定仕訳（is_realized=false）が含まれる
   科目・取引先名が確定後に変わる可能性あり
```

### 翌月の同期で自動更新

```
仕訳IDでUPSERT:
  未確定 → 確定に変わったら上書き
  科目・取引先名が変わったら上書き
  レポートは次回同期時に再生成
```

## 4. レート制限と同期戦略

### MF MCPのレート制限（34_md セクション2-2）

```
制限: 1,000リクエスト/時（1トークンあたり）
```

### 1社あたりの同期に必要なリクエスト数

| API | リクエスト数 | 備考 |
|---|---|---|
| getJournals | 1〜5 | ページネーション（100件/ページ） |
| getAccounts | 1 | 科目マスタ（108〜133件。課税方式で変わらない。2026-05-23検証済み） |
| getTaxes | 1 | 税区分マスタ（151件。available件数は課税方式で変動: 0〜78件） |
| getTermSettings | 1 | 会計年度設定（tax_method（課税方式）を含む。税区分フィルタに必要） |
| getReportsTrialBalanceProfitLoss | 1 | 今期分 |
| getReportsTransitionProfitLoss | 1 | 今期分 |
| getDataSources | 1 | 口座/カード |
| getTradePartners | 1 | 取引先マスタ |
| **合計** | **8〜12** | 平均10リクエスト/社 |

### 直列 + 3社並列

```
200社 × 10リクエスト = 2,000リクエスト

3社並列 × 10リクエスト = 30リクエスト/バッチ × 2秒間隔
= 900リクエスト/時（制限1,000の90%）
= 約22分で200社完了
```

| 方式 | リクエスト/時 | 所要時間 | レート制限 |
|---|---|---|---|
| 全並列 | 2,000/瞬間 | 数秒 | ✗ 即死 |
| 10並列 | 2,000/数分 | 5分 | ✗ 超過リスク |
| 直列（1社ずつ） | 600/時 | 約60分 | ✓ 安全 |
| **直列 + 3社並列** | **900/時** | **約22分** | **✓ 安全（90%）** |

### エラーハンドリング

```
1社失敗 → ログに記録して次の社に進む（全体を止めない）
リトライ → 1社につき最大3回、30秒間隔
全社完了後 → 失敗リストをレポートに含める
```

## 5. 同期の実行基盤

### Supabase Edge Functions + pg_cron

```
pg_cron: 毎月20日 0:00 → sync_queueテーブルに200社分のジョブを投入
Edge Function: 10秒ごとにキューから3社取り出して並列同期実行
完了: 全社終わったらレポート生成
```

> Edge Functionsの実行時間上限（150秒/リクエスト）があるため、1回のFunction呼び出しで3社を処理。
> キューが空になるまで繰り返し。

## 6. レポート自動保管

### 月次同期後に全パターンを自動実行

```
同期完了後:
  全顧問先 × 対象パターン を自動実行 → 結果をDBに保存

保存テーブル: mf_monthly_reports
  | カラム | 型 | 説明 |
  | client_id | TEXT | 顧問先ID |
  | pattern_id | TEXT | 'p01_sales_ranking' 等 |
  | fiscal_year | INT | 2025 |
  | report_month | INT | 12（同期対象月） |
  | report_json | JSONB | ランキング結果（構造化データ） |
  | created_at | TIMESTAMPTZ | レポート生成日時 |
```

### 差分レポート（新規検出）

同期時に前回との差分を自動検出し、レポートとして保管:

```
テーブル: mf_diff_reports
  | client_id | diff_type | diff_json | report_month |

diff_type:
  'new_trade_partners'  → 新しい取引先
  'new_bank_accounts'   → 新しい口座
  'new_credit_cards'    → 新しいカード
```

### レポート出力

```
Phase 1: UI画面（Vue）
  → 顧問先ごとのランキング・増減・差分をVue画面で表示
  → フィルタ（期間、パターン種別）、ソート、ドリルダウン対応

Phase 2: PDF出力ボタン（必要時のみ）
  → UI画面に「PDF出力」ボタンを追加
  → 表示中のレポートをPDF化して保存/ダウンロード
```

> **通知機能はまだ不要。** sugu-sruに通知機能はあるが、レポートUI完成後に検討。

## 7. 部品アーキテクチャ

### 方針: 今MCP版で作り、DB移行時にfetch*の内部だけ差し替える

> **Repository層パターンと同じ考え方。**
> 部品のインターフェース（入出力の型）を固定すれば、
> 呼び出し側（コマンドレシピ）はデータソース変更の影響を受けない。
> 部品定義の詳細は [35_parts_catalog.md](35_parts_catalog.md) を参照。

```
今:       fetchJournals(token, opts) → 内部でMCP呼び出し → Journal[] を返す
DB移行後: fetchJournals(token, opts) → 内部でDBクエリ   → Journal[] を返す
                                        ↑ここだけ差し替え
```

| パーツ種別 | DB移行時の影響 |
|---|---|
| `fetch*`系（データ取得） | 内部実装をMCP→DBに差し替え |
| `aggregate*`系（集計） | **変更なし**（純粋なロジック） |
| `nayose`（名寄せ） | **変更なし**（既に分離済み） |
| `formatRanking`（出力） | **変更なし**（純粋なフォーマット） |
| `calcChange`（増減算出） | **変更なし**（純粋な算出） |

### ファイル構成

```
src/ai-commands/
  engine/
    parts/
      fetchAccounts.ts          ← 科目取得+フィルタ（現在MCP、DB移行時に内部差し替え）
      fetchJournals.ts          ← 仕訳取得（現在MCP、DB移行時に内部差し替え）
      fetchPL.ts                ← PL試算表/推移表取得（現在MCP、DB移行時に内部差し替え）
      fetchDataSources.ts       ← 口座/カード取得（現在MCP、DB移行時に内部差し替え）
      aggregateByPartner.ts     ← 取引先別集計（データソース非依存）
      aggregateByDepartment.ts  ← 部門別集計（データソース非依存）
      aggregateByMonth.ts       ← 月別集計（データソース非依存）
      nayose.ts                 ← 名寄せ（データソース非依存。新規のみAI）
      calcChange.ts             ← 増減算出（データソース非依存）
      formatRanking.ts          ← テーブル出力（データソース非依存。⚠未確定マーク対応）
      aiSummarize.ts            ← AI要約（データソース非依存）
    sync/
      monthlySync.ts            ← 月次同期バッチ（DB移行後に実装）
      diffDetector.ts           ← 差分検出（DB移行後に実装）
      reportGenerator.ts        ← レポート自動生成+保管（DB移行後に実装）
  patterns/
    p01_sales_ranking.ts        ← パーツ組み合わせ（5〜10行）
    p02_expense_ranking.ts
    p22_sales_change_ranking.ts
    ...
```

### パターン→パーツ対応（全28パターン）

| グループ | パターン（個数） | 共通パーツ |
|---|---|---|
| **A: ランキング系** | #1,#2,#5,#22-#28（11個） | fetchAccounts + fetchJournals + aggregateByPartner + nayose + formatRanking |
| **B: 月次推移系** | #3,#6,#7（3個） | fetchJournals + aggregateByMonth |
| **C: PL概況系** | #4,#9（2個） | fetchPL + aiSummarize |
| **D: データソース系** | #8,#15,#19,#21（4個） | fetchDataSources |
| **E: 消込系** | #11-#14（4個） | fetchJournals + 独自ペアリング |
| **F: 単独系** | #10,#16-#18,#20（4個） | 個別ロジック |

> **共通パーツで20/28パターン（71%）をカバー。**
