## 2. 全設計決定事項

### 2-2. AIが使えるツールの範囲

**MFツール19個 + sugu-sru内部API** の両方を含む。

#### MF MCPツール（19個）

READ 15個 + WRITE 4個。全ツールの詳細スキーマは [37_infra_mcp.md セクション14-5](37_infra_mcp.md) を参照。

#### sugu-sru内部API

| API | 種別 | 用途 |
|---|---|---|

| 顧問先詳細（`clientRoutes`） | READ | 顧問先の基本情報・契約情報 |
| 見込先一覧（`leadRoutes`） | READ | 見込先の一覧・詳細 |

### 2-6. AIのトーン

**丁寧だが堅すぎない。AI初心者のスタッフが安心して使えるレベル。**

```
❌ NG（堅すぎ）: 「お問い合わせいただきありがとうございます。以下の結果をご報告申し上げます。」
❌ NG（ぞんざい）: 「はい、やった。結果これ。」
✅ OK: 「テスト法人の勘定科目を取得しました。全45科目あります。
         気になる科目があれば、科目名で絞り込めますよ。」
✅ OK: 「すみません、その指示はちょっとわかりませんでした。
         たとえば「○○社の仕訳を見せて」のように指示してみてください。」
```

**禁止事項:**
- 想像・推測で回答しない。わからないときは正直に「わかりません」と言う
- データがない場合に架空のデータを返さない
- エラー時に原因を推測しない（事実のみ報告）

### 2-7. コスト管理

| 項目 | 仕様 |
|---|---|
| 設定場所 | `http://localhost:5173/#/admin-dashboard?tab=cost` |
| **分類** | **仕訳操作**（画像AIOCR → 仕訳作成の一連フロー）と **質問・MCP**（READ/WRITE問わずMCPへの問い合わせ・操作全般）の2分類 |
| 上限単位 | **月額上限**（円 or リクエスト数） |
| 上限超過時 | AIコマンドの実行を停止し「今月の利用上限に達しました」と表示 |
| ダッシュボード表示 | 今月の利用状況（グラフ）、残りの利用可能数、利用履歴 |

> **注意**: 仕訳操作とMCP質問は別カウント。仕訳操作の上限に達してもMCP質問は使える（逆も同様）。

### 2-8. エラーハンドリング

| エラー種別 | 対応 |
|---|---|
| **システムエラー**（MCP接続失敗、API通信エラー等） | 管理者にコピペ可能なエラーコード・ログを表示。「このエラーコードを管理者に共有してください: `ERR-MCP-403-20260520T1130`」 |
| **AI解析不能**（意図がわからない） | 「すみません、その指示はわかりませんでした。こんな指示ならできます：」+ 候補を提示 |
| **データなし** | 「該当するデータが見つかりませんでした。」（想像で埋めない） |

### 2-9. 操作ログ（2つの役割）

操作ログには**2つの独立した目的**がある：

#### 役割1: 再実行可能ログ（ナレッジ共有）

ログ一覧からクリックすると**同じ操作を再実行**できる。
「あの人がやっていた操作を自分もやりたい」「この前の集計をもう一度」に対応。
スタッフ全員の操作ログが閲覧可能（会話履歴と同様にナレッジベース的活用）。

#### 役割2: WRITE操作の修復ログ（安全網）

**Gemini 2.5はミスする**。WRITE操作（仕訳作成・取引先登録等）の結果を詳細に記録し、
ミスが発覚した場合に**修復（取消・差戻し）できるようにする**。

| 記録項目 | 内容 |
|---|---|
| タイムスタンプ | 実行日時 |
| ユーザー | 実行したスタッフ名 |
| 入力テキスト | ユーザーの自然言語入力 |
| 解析結果 | AIが判定したツール名・パラメータ |
| 実行結果 | 成功/失敗・レスポンス要約 |
| WRITE操作の場合 | 確認モーダルの承認/拒否 |
| **WRITE結果の詳細** | **作成された仕訳ID・取引先コード等。修復時に使用** |
| **WRITE前の状態** | **更新操作の場合、変更前のデータスナップショット** |
| 再実行ボタン | ✅ ログからワンクリックで同じ操作を再実行 |
| 修復ボタン | ✅ WRITE操作の場合、取消・差戻しを実行 |

### 2-10. コンテキスト認識（AIが「今何を見ているか」を理解する）

AIは以下の4つのコンテキストを認識して操作・回答できる：

| コンテキスト | 取得方法 | 使用例 |
|---|---|---|
| **現在のページ** | Vue Routerの`route.path` + `route.params` | 「このページの顧問先のMF状態を確認」→ 顧問先詳細ページなら自動でclientIdを取得 |
| **表示されているUI** | ページコンポーネントが公開するデータ（テーブルの行数、表示中の仕訳一覧等） | 「この一覧をCSVにして」→ 表示中のテーブルデータを取得 |
| **システム全体** | sugu-sru内部API（全顧問先、全仕訳等） | 「全顧問先のMF連携状態を確認」→ 全体を横断して取得 |
| **ユーザーが選択したテキスト** | `window.getSelection()` | テキスト選択 → 右クリック「AIに聞く」→ 選択部分を質問に含める |

コンテキストの優先順位：**選択テキスト > 現在のページ > システム全体**

これにより「この顧問先の」を省略して「仕訳見せて」だけで動く。

### 2-11. AIによる提案（カスタムコマンドの代替）

ユーザーがコマンドを定義するのではなく、**AIが「できること」を提案する。**

```
ユーザー: 「テスト法人について何かわかる？」

AI: 「テスト法人（株式会社すぐする）について、以下のことができます：

  [勘定科目を確認]  [今期の仕訳一覧]  [決算月の整合チェック]
  [税区分を確認]    [取引先一覧]      [残高試算表を取得]

  気になるものをクリックするか、自由に質問してください。」
```

ボタンクリック or 自然言語入力の両方で実行できる。

### 2-12. 2モード設計（AIの裁量最小化）

> **大前提: AIはブレる。嘘をつく。**
> だからAIの裁量を最小化する。AIには「結果を日本語にまとめる」だけやらせる。

#### 定型対応（本命）

- **パターンマッチ**でユーザーの意図をコード側で分類
- **ツール名・引数はコードで確定**（AIに聞かない）
- AIは「MCP実行結果のJSON → 人間が読める日本語」にするだけ
- AIの裁量ゼロ → ブレようがない

```
ユーザー: 「先月の仕訳を見せて」

sugu-sru（コード）:
  ① パターンマッチ → journals_period にヒット
  ② 日付計算 → start_date: "2026-04-01", end_date: "2026-04-30"
     ※ AIではなくコードが計算
  ③ mfc_ca_getJournals({ start_date, end_date }) を実行
  ④ 結果JSONを取得

AI（要約だけ）:
  「4月の仕訳は12件です。
   | 日付 | 借方 | 貸方 | 金額 | 摘要 |
   | 4/1 | 旅費交通費 | 事業主借 | 500 | ... |
   ...」

→ AIはツール選定にも引数にも関与していない。要約しただけ。
```

#### Layer 1 定型パターン一覧

> データ取得系コマンド（Layer 1パターンマッチ）の詳細は [34_command_catalog.md](34_command_catalog.md) の「データ取得系」セクションに移動。
> 本セクションでは処理フロー（Layer 1/2/3）の概要のみ記載。

```
処理フロー（3層構造）:
  ユーザー入力
    ↓
  Layer 1: コード側パターンマッチ（34_command_catalog.md データ取得系）
    「科目一覧」→ accounts_list → 即座にMCP実行 → 表示
    「先月の仕訳を見せて」→ journals_period → 即座にMCP実行 → 表示
    ↓ マッチしない場合
  Layer 2: AIルーティング（本セクション）
    「銀行明細を仕訳して」→ 銀行/カード仕訳候補 → 部品組み合わせで実行
    「売上トップ10は？」→ 売上ランキング → 部品組み合わせで実行
    ↓ それもマッチしない場合
  Layer 3: オープン対応（フォールバック）
    AIにFunction Callingの裁量を与える
```

#### Layer 2: AIルーティング

> ルーティングは部品ではなくフロー制御。コマンドを選ぶ行為。

| 項目 | 内容 |
|---|---|
| 目的 | 自然言語入力をコマンドに振り分ける |
| 入力 | ユーザーの自然言語テキスト |
| 出力 | JSON: `{"command": "コマンド名"}` |
| **モデル** | **gemini-3.5-flash**（決定: 2026-05-22） |
| コスト | ~¥0.001/回（入力~415トークン + 出力~10トークン） |

```
やること:
  ✅ コマンドカタログを読み、最適なコマンドを返す
  ✅ 該当なしの場合は "unknown" を返す

やってはダメ:
  ❌ カタログに存在しないコマンドを生成しない
  ❌ コマンドをスキップして直接処理しない
  ❌ ユーザーの意図を推測で補完しない
```

紛らわしいコマンドの境界定義:

| 曖昧性 | 判定基準 |
|---|---|
| 銀行仕訳 vs 領収書仕訳 | 「銀行」「カード」→ 銀行/カード / 「領収書」「レシート」→ 領収書 |
| 経費ランキング vs 月次変動 | 「ランキング」→ 経費ランキング / 「増えた」「変動」→ 月次変動 |
| 仕訳投入 vs 仕訳取消 | 「登録」「投入」→ 投入 / 「修正」「取消」→ 取消 |

ルーティングテスト結果:

| モデル | 正解率 | 平均ms | 判定 |
|---|---|---|---|
| gemini-2.5-flash | 20/20 (100%) | 1731ms | ✅ 採用可 |
| gemini-3-flash-preview | 19/20 (95%) | 5113ms | ✅ 採用可 |
| gemini-3.1-flash-lite | 20/20 (100%) | 2165ms | ✅ 採用可 |
| **gemini-3.5-flash** | **20/20 (100%)** | **3067ms** | **✅ 採用（決定）** |

テストスクリプト: [test_routing.ts](../../src/scripts/test_routing.ts)

> **決定: gemini-3.5-flash を採用。** 1万回呼んでも¥10。コストは無視できる。


#### オープン対応（フォールバック）

- 定型パターンにマッチしなかった場合**のみ**発動
- **やむを得ない**のでAIにFunction Callingの裁量を与える
- AIに設計書の関連セクションを検索させて（`search_design_docs`）、自分でツールを選ばせる
- ブレるリスクがあるが、「対応できません」より良い

```
ユーザー: 「不動産の利回りは適正か？」

sugu-sru:
  ① パターンマッチ → マッチなし → オープン対応
  ② AIに以下を渡す:
     - ユーザーの質問
     - search_design_docs ツール（設計書検索）
     - MCPツール19個のFunction Calling定義
     - テナント情報

AI（自律判断）:
  Step 1: search_design_docs("不動産 利回り") → 設計書を読む
  Step 2: mfc_ca_getReportsTrialBalanceProfitLoss() → PL取得
  Step 3: 賃貸料 ÷ 不動産関連経費 で分析
  Step 4: 回答

→ AIが自律的に判断。正しいかもしれないし、間違うかもしれない。
```

#### AIの裁量の範囲まとめ

| 段階 | 定型対応 | オープン対応 |
|---|---|---|
| 意図分類 | **コード** | **AI** |
| ツール選定 | **コード** | **AI** |
| 引数生成 | **コード** | **AI** |
| MCP実行 | **コード** | **コード** |
| 結果の要約 | **AI**（日本語にする） | **AI**（日本語にする） |
| 精度 | ✅ ブレない | ⚠️ ブレるかも |

> **本命は定型対応。** 定型パターンを充実させればさせるほど、システムの信頼性が上がる。
> オープン対応は「おまけ」。あると便利だが、これに依存してはいけない。

### 2-13. テナント解決

> **MF MCPに`select_tenant`ツールは存在しない。**
> MCPは「1つのOAuthトークン = 1つの事業所」で固定。
> テナント切替はMCPではなくアプリ側の責務。
> 詳細は [37_infra_mcp.md セクション7](37_infra_mcp.md) を参照。

```
ユーザー: 「株式会社すぐするの4月試算表を出して」

sugu-sru:
  ① 「株式会社すぐする」→ clients.json（将来Supabase）検索
  ② → clientId: c_XXXXXXXX を特定
  ③ → mfAuthService.getValidAccessToken(clientId) でOAuthトークン取得
  ④ → そのトークンでMCPクライアント接続
  ⑤ → 以降は通常のMCPツール呼び出し

コンテキストからの自動解決:
  - 顧問先詳細ページにいる → route.params.clientId から自動解決
  - チャット内で「この顧問先」→ 直前のコンテキストから解決
  - 曖昧な場合 → 「どの顧問先ですか？」と聞き返す
```

### 2-14. 英日辞書（`mfc_ca_en_ja_dictionary`）の活用

MF MCPには全ツールの説明に `Use 'mfc_ca_en_ja_dictionary' tool if you need Japanese translation.` と記載されている。
**MFはAIが最初にこの辞書ツールを呼ぶことを想定している。**

| 項目 | 仕様 |
|---|---|
| 呼び出しタイミング | テナント初回接続時に1回。結果をキャッシュ |
| キャッシュ有効期間 | 1日（テナントごと） |
| 用途 | APIレスポンスの英語キー → 日本語変換。AIの要約精度向上 |
| 定型対応での使い方 | コード側で辞書を参照し、AIに渡す結果JSONのキーを日本語に変換 |
| オープン対応での使い方 | AIのシステムプロンプトに辞書テキストを含めて理解を助ける |

## 3. WRITE確認モーダル — 設計（汎用）

> **仕訳だけではない。** 取引先作成、明細登録、データ更新など全WRITE操作に適用する汎用設計。

### 決定: 案C（チャット内インライン）ベース + 複合データは案B（テーブル）に自動切替

#### パターン1: 単純操作（取引先作成等）

```
AI: 取引先を新規登録します。よろしいですか？

  🏢 取引先情報
  ├── 名称: 山田商事株式会社
  ├── 法人番号: 1234567890123
  └── インボイス番号: T1234567890123

  [✅ 登録する]  [✏️ 修正する]  [❌ やめる]
```

#### パターン2: 仕訳作成（単純仕訳）

```
AI: 仕訳を作成します。よろしいですか？

  📋 テスト法人 — 2026/05/20
  ├── 借方: 消耗品費 ¥1,500（課税仕入10%）
  └── 貸方: 現金 ¥1,500
  摘要: 文具購入

  [✅ 作成する]  [✏️ 修正する]  [❌ やめる]
```

#### パターン3: 複合仕訳・複数行（テーブル自動切替）

```
AI: 複合仕訳を作成します。よろしいですか？

  📋 テスト法人 — 2026/05/20

  │ 借方          │ 金額    │ 貸方     │ 金額    │ 摘要       │
  │ 消耗品費      │ ¥1,500  │ 現金     │ ¥3,000  │ 文具・備品 │
  │ 事務用品費    │ ¥1,500  │          │         │            │

  ▼ 詳細を表示（API送信パラメータ）

  [✅ 作成する]  [✏️ 修正する]  [❌ やめる]
```

#### パターン4: データ更新（変更前後の差分表示）

```
AI: 仕訳を更新します。よろしいですか？

  📋 仕訳 #42 — テスト法人

  │ 項目     │ 変更前        │ 変更後        │
  │ 借方科目 │ 消耗品費      │ 事務用品費    │
  │ 金額     │ ¥1,500        │ ¥2,000        │

  ⚠️ この操作は取り消せます（操作ログから修復可能）

  [✅ 更新する]  [✏️ 修正する]  [❌ やめる]
```

#### パターン5: 一括操作（複数件同時）

```
AI: 5件の仕訳を一括作成します。よろしいですか？

  │ # │ 日付       │ 借方     │ 貸方 │ 金額    │
  │ 1 │ 2026/05/01 │ 消耗品費 │ 現金 │ ¥1,500  │
  │ 2 │ 2026/05/03 │ 交際費   │ 現金 │ ¥5,000  │
  │ 3 │ 2026/05/10 │ 旅費交通 │ 現金 │ ¥1,200  │
  │ ...残り2件 |

  [全件表示]  [✅ 5件すべて作成]  [1件ずつ確認]  [❌ やめる]
```

### 自動切替ルール

| 条件 | 表示パターン |
|---|---|
| フィールド3個以下（取引先作成等） | パターン1（ツリー表示） |
| 仕訳1行（単純仕訳） | パターン2（ツリー表示） |
| 仕訳2行以上（複合仕訳） | パターン3（テーブル表示） |
| 更新操作（既存データ変更） | パターン4（差分表示） |
| 複数件一括操作 | パターン5（一覧テーブル） |

## 5. AIの使い方の具体例（仕訳以外）

> 参考: [「仕訳を作る」だけじゃもったいない — マネフォMCPサーバーのもう一つの活用法](https://note.com/kyon_copanda/n/ne1a9a90ccfb8)

### 5-1. 経営者向け即答

```
ユーザー: 「今月の販管費はいくら？」
AI: テスト法人の2026年5月の販管費を集計します...

  📊 販管費合計: ¥1,234,567
  ├── 人件費: ¥800,000
  ├── 消耗品費: ¥150,000
  ├── 交際費: ¥80,000
  └── その他: ¥204,567

  前月比: +12.3%（4月: ¥1,098,765）

  [詳細内訳を見る]  [CSV出力]
```

### 5-2. 画面コンテキスト活用

```
ユーザー:（顧問先一覧ページで）「この一覧でMF未連携の顧問先は？」
AI: 表示中の14件の顧問先のうち、MF未連携は3件です。

  │ 顧問先名   │ MF連携状態 │
  │ ○○商事    │ ❌ 未連携   │
  │ △△工業    │ ❌ 未連携   │
  │ □□サービス │ ❌ 未連携   │

  [MF連携を開始]  [一覧をCSV出力]
```

### 5-3. テキスト選択 → 質問

```
ユーザー:（仕訳一覧で「事業主借」を選択 → 右クリック「AIに聞く」）
AI: 「事業主借」は個人事業で使う勘定科目です。
  個人のお金を事業に入れた場合に使います。

  テスト個人の事業主借の残高: ¥350,000（今期）

  [事業主借の仕訳一覧を見る]  [事業主貸との差額を確認]
```

### 5-4. データ整合チェック（sugu-sru ↔ MF）

```
ユーザー: 「MFとデータ整合チェックしろ」
AI: 全顧問先のMFデータとsugu-sruの整合性をチェックします...

  ✅ テスト法人: 決算月一致（6月）、科目数一致
  ⚠️ テスト個人: 決算月不一致
     sugu-sru: 11月 → MF: 12月
     [修正する]
  ⚠️ テスト個人: 仕訳件数差異
     sugu-sru: 42件 → MF: 45件（3件多い）
     [差分を確認する]

  [全詳細レポート]  [不一致のみ表示]
```
