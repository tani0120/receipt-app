# 第Ⅰ部：非対称性起点の機能網羅マップ

## 0. 大前提：課題の本質

顧問先・税理士・独自システムという3主体の間で、「持っている情報・機能」と「持っていない情報・機能」の分布が一致していない。

| 主体 | 持っている情報 | 持っていない情報 | 持っている機能 | 持っていない機能 |
|------|-------------|----------------|-------------|----------------|
| 顧問先 | 取引の目的・背景（事実） | 会計知識・勘定科目の知識 | （なし） | 仕訳変換、過去データ照合 |
| 税理士 | 会計知識・勘定科目の知識 | 取引の目的・背景（事実）、個別の過去処理の記憶 | （なし、属人的記憶に依存） | 過去データの自動照合、大量データの一括処理 |
| 独自システム | 同期・蓄積された過去仕訳データ | 取引の目的・背景（事実そのもの） | 過去仕訳照合、AI仕訳変換、資料整合性チェック | 事実の取得（顧問先からの入力なしには動けない） |

**非効率の正体**：この分布のズレが噛み合わないまま、メール・電話による都度確認という非効率なプロセスで補い合っていたこと。

### 設計原則

> 各主体に、その主体が「持っていない情報・機能」を求めない

- 顧問先には「事実」だけを求める（会計知識・機能を要求しない）
- 独自システムには「事実から勘定科目への変換」「過去仕訳との照合」を担わせる
- 税理士には「変換結果の確認」だけを求める（確認が必要な分のみ）

### 税理士と独自システムを分離する理由

独自システムは税理士（事務所）が所有・運用するものだが、**所有関係と保有情報・機能の分布は別軸**であるため、行を統合しない。

- このシステムを作る発端は「税理士個人の記憶は属人的で、担当が変わると失われる」という課題
- 税理士行と独自システム行を統合すると、「税理士は過去仕訳データも当然持っている」という誤った前提を生み、属人性の問題（システムの存在理由そのもの）が表から消えてしまう

## 1. MECE分解の構造

非対称性を「誰と誰の間か（主体の組）」×「何が非対称か（種類）」でMECEに分解し、各セルに対応する機能・対応状況を当てる。

### 主体の組（4種）

| 組 | 内容 |
|---|---|
| A. 顧問先 ↔ 税理士 | 最初に発見した、最大の非対称 |
| B. 顧問先 ↔ 独自システム | 顧問先の事実をシステムがどう受け取るか |
| C. 税理士 ↔ 独自システム | システムが処理したことを税理士がどう検証するか |
| D. 税理士（個人）↔ 税理士（事務所・後任） | 属人性の問題。引き継ぎの非対称 |

### 非対称の種類（5種）

| 種類 | 内容 |
|---|---|
| ①事実の非対称 | 取引の背景・目的を誰が知っているか |
| ②知識の非対称 | 会計・税務の専門知識を誰が持っているか |
| ③データの非対称 | 過去の処理記録を誰が即座に引き出せるか |
| ④状態の非対称 | 「今何が起きているか」（資料の有無・進捗）を誰が把握しているか |
| ⑤認識の非対称 | 処理結果が正しいかどうかを、誰がどこまで認識・検証できるか |

4組 × 5種 = 20セル。該当しないセルは「対象外」として明記し、ダブりを避ける。

## 2. A：顧問先 ↔ 税理士

### A-① 事実の非対称（核）

顧問先：取引の目的・背景を知っている
税理士：知らない（メール・電話で都度確認していた）

| 機能 | 内容 | 実装詳細 |
|---|---|---|
| 4区分の確認フォーム | 仕入／経費／資金移動／資産増加（入金側：売上／借入／資金移動／その他収入）を、機械判定できない場合のみ聞く | → 第Ⅱ部§8 |
| 顧問先向け文言の言い換え | 会計用語を使わず、事実ベースの表現に変換（「立替」→「自分が会社・事業のお金を使った」等） | → 第Ⅱ部§8 |
| アコーディオン形式 | 選択肢名は常に表示、説明・例文はタップで展開し、迷いを即時解消 | → 第Ⅱ部§8 |

### A-② 知識の非対称（核）

税理士：会計・税務知識を持っている
顧問先：持っていない

| 機能 | 内容 | 実装詳細 |
|---|---|---|
| AI仕訳変換 | 顧問先の事実回答（4区分）を制約として、AIが具体的な勘定科目を選択。顧問先には会計知識を要求しない | → 第Ⅱ部§11 |
| 「科目は何でも良い」原則 | 4区分・税区分が合えば、科目の細部は厳密に当てなくてよい（税理士の確認コストを最小化） | — |
| 顧問先への結果非開示 | AI仕訳案・処理結果は顧問先に見せない（判断できない情報を渡さない） | → 第Ⅱ部§11 |

### A-③ データの非対称 → 対象外

税理士の属人的記憶の問題であり、C系（税理士↔システム）およびD系（属人性）に帰着する。顧問先側には対応する機能は不要。

### A-④ 状態の非対称

顧問先：資料を出したかどうかを把握していないことがある
税理士：資料が来ているか、漏れているかを毎回確認する手間がある

| 機能 | 内容 | 実装詳細 |
|---|---|---|
| チェックリスト（状態変化） | 取引データに現れる前の事実（口座追加・雇用等）を先に拾う | — |
| チェックリスト（今期・来期の予定） | 税額予測・簡易/本則課税判定に使う将来情報を拾う | — |
| 証拠書類の添付依頼（理由A・B） | 初回契約系・内訳系のトリガーに応じて、必要な場合だけ任意で依頼 | → 第Ⅱ部§9 |

### A-⑤ 認識の非対称

顧問先：処理結果が正しいかを判断できない
税理士：処理結果が税額に影響するかを判断できる

| 機能 | 内容 | 実装詳細 |
|---|---|---|
| 処理結果の非開示（確定方針） | 顧問先に結果を見せて「合っているか」を確認させる設計は撤回済み | — |
| 教育コンテンツ・選択肢改善ループ | 検討したが撤回。会計知識を求めない原則に反する／税額に影響しない領域への過剰投資 | 対象外§15 |

## 3. B：顧問先 ↔ 独自システム

### B-① 事実の非対称 → A-①と同一

4区分確認フォームで対応済み。ダブりのため新規機能なし。

### B-② 知識の非対称 → 対象外

顧問先に機能を持たせる設計は不要（設計原則どおり）。

### B-③ データの非対称

独自システム：MF連携済みのデータは保有する
独自システム：未連携の口座・カードのデータは保有しない
顧問先：未連携の口座・カードの存在を知っている

| 機能 | 内容 | 実装詳細 |
|---|---|---|
| MF連携済み一覧の提示＋差分確認 | 顧問先に一覧を見せ「これで全部か」を確認させ、漏れを発見する | → 第Ⅲ部§16-1 |
| 未連携口座・カード管理テーブル | 登録日・最終受領日・受領方法・受領の安定性を記録 | → 第Ⅲ部§16-1 |

### B-④ 状態の非対称

顧問先：資料を出したかどうかを意識していない
独自システム：最後にいつ受領したかは記録できるが、出されなければ検知できない

| 機能 | 内容 | 実装詳細 |
|---|---|---|
| 資料不足検知 | 「ページを開いた日時 − 最終受領日時 ＞ 閾値」で、未連携口座・カード／領収書の不足をアラート表示 | → 第Ⅲ部§16-2 |
| 確認・確定フロー不要の原則 | 参考情報のため、検知結果に承認フローは設けない | — |

### B-⑤ 認識の非対称 → A-⑤と同一方針（非開示）

ダブりのため新規機能なし。

## 4. C：税理士 ↔ 独自システム

### C-① 事実の非対称 → 既存設計に内包

A-①の確認フォームのデータが、そのまま税理士に渡る経路。新規機能なし。

### C-② 知識の非対称

税理士：会計・税務の専門知識を持つ
独自システム：知識は持たないが、過去の処理結果のパターンマッチングはできる

| 機能 | 内容 | 実装詳細 |
|---|---|---|
| AI仕訳変換エンジン | 画像＋4区分の制約内で科目を推論し、税理士が確認しやすい形で提示 | → 第Ⅱ部§11 |
| 固定マッピングエンジン | 仕入・立替・資金移動・資産増加・借入は、区分が決まれば科目も自動確定（AI推論不要） | → 第Ⅱ部§11 |

### C-③ データの非対称（核）

独自システム：MFから同期した過去仕訳データを保有する
税理士：個人の記憶だけでは過去処理を保持できない（属人的、引き継ぎで失われる）

| 機能 | 内容 | 実装詳細 |
|---|---|---|
| 過去仕訳照合エンジン | 取引先名×4区分×税区分が過去と一致するかを機械的に判定し、自動確定 | → 第Ⅱ部§5 |
| データ同期（自動＋イベント駆動） | MCP経由の定期同期と、顧問先保存時の即時同期の2系統 | → 第Ⅱ部§4 |
| 「特定できる」と「自動化できる」の区別 | 人間が判断可能でも、システムが実装済みのロジックでカバーできる範囲だけを自動化対象とする | — |

### C-④ 状態の非対称

独自システム：取引データ・添付資料の到着状況を保持する
税理士：何が確認待ちか、何が要修正かを都度見ないと分からない

| 機能 | 内容 | 実装詳細 |
|---|---|---|
| 税理士向け確認画面（要確認フラグ） | 資料不足・整合性NG・税区分混在の場合にフラグを立てる | → 第Ⅱ部§12 |
| 口座・カードプロファイル | 主な取引先・入金/支払サイト・定期支払の有無等を集計し、異常検知の文脈情報として提示 | → 第Ⅲ部§16-3 |
| プロファイル更新トリガー | 月末（定期）＋ページを開いたタイミング（イベント駆動）で再計算、確定フロー不要 | — |

### C-⑤ 認識の非対称

独自システム：処理の根拠（店名・資料・回答内容）を保持する
税理士：根拠を見ないと、AIの判断が妥当かを認識できない

| 機能 | 内容 | 実装詳細 |
|---|---|---|
| AI仕訳案＋推論根拠の提示 | 画像のどの部分から判断したかを併記し、税理士が判断材料を持てるようにする | → 第Ⅱ部§10, §12 |
| 最終承認フロー | 税額に影響する処理は、必ず税理士の承認を経てMFに登録される | → 第Ⅱ部§12 |

## 5. D：税理士（個人）↔ 税理士（事務所・後任）

### D-① 事実の非対称 → 既存設計の副次効果

A-①で集めた事実（4区分回答・添付・メモ）が独自システムDBに記録されることで、自動的に解消される。新規機能なし。

### D-② 知識の非対称 → 対象外

会計知識は事務所内で共有されている前提（個人差はあるが、本質的にシステムが埋める対象ではない）。専門知識の引き継ぎは本システムの対象範囲ではない。

### D-③ データの非対称（核）

担当税理士：個別の過去処理を記憶している（属人的）
後任・事務所：記憶を共有されておらず、引き継ぎで失われる

| 機能 | 内容 | 実装詳細 |
|---|---|---|
| 過去仕訳照合DB（蓄積） | 取引先名×4区分×科目の組み合わせをDBに記録し、担当が変わっても同じ判定が継続する | → 第Ⅱ部§5, §13 |
| 用語の正確な定義（「学習」の排除） | 「過去仕訳照合（DB蓄積型）」と呼び、AIモデルの学習ではなく単純な記録・検索であることを明示 | → 第Ⅱ部§5 |
| 口座・カードプロファイルの「入手方法」記録 | 「この顧問先はこの口座をこう資料でもらう」という運用ノウハウをDBに記録し、属人性を排除 | → 第Ⅲ部§16-3 |

### D-④ 状態の非対称 → 既存設計の適用

C-④の「税理士向け確認画面」「資料不足検知」がそのまま事務所全体でも機能する。新規機能なし。

### D-⑤ 認識の非対称 → 既存設計の副次効果

C-⑤の「AI仕訳案＋推論根拠の提示」が、後任への引き継ぎ情報としても機能する。新規機能なし。

## 6. 検討したが対象外と判断したもの

| 案 | 該当する非対称 | 対象外の理由 |
|---|---|---|
| 税額・資金繰り予測 | advisory的だが非対称解消ではない | 目的（情報の非対称性の解消）から外れる。予測は別の課題（経営助言） |
| 取引先の請求・支払漏れ検知 | 経営上の異常検知 | 情報の非対称性ではなく、経営管理の課題。目的外 |
| 事務所全体ダッシュボード | D組に見えるが実態は業務管理 | 顧問先↔税理士の非対称ではなく、事務所の内部業務効率化。目的外 |
| 教育コンテンツ・選択肢改善ループ | A-⑤に見えたが実際は知識を学ばせる設計 | 顧問先に会計知識を求めない原則に反する。科目の精密化は税額に影響せず費用対効果が悪い |
| 制度改正の影響事前通知 | 外部情報との接続 | 内部データのみを扱う設計の対象外。advisory機能 |
| 協力度合いの定量化・料金反映 | A組の派生に見えたが無関係 | 既存の定量化済みの仕組みと無関係。税理士の感覚的判断で代替可能であり意義が薄い |

## 7. 網羅性の検証（20セルの埋まり方）

|  | ①事実 | ②知識 | ③データ | ④状態 | ⑤認識 |
|---|---|---|---|---|---|
| **A 顧問先↔税理士** | 4区分確認・言い換え（核） | AI仕訳変換・科目何でも良い原則（核） | 対象外（C系に帰着） | チェックリスト・証拠書類依頼 | 非開示方針（教育案は撤回） |
| **B 顧問先↔システム** | A-①と同一（ダブりなし） | 対象外（機能不要） | MF連携一覧確認・未連携管理 | 資料不足検知 | 対象外（A-⑤と同一） |
| **C 税理士↔システム** | A-①データの受け渡し（既存内包） | AI仕訳変換・固定マッピング | 過去仕訳照合・データ同期（核） | 確認画面フラグ・口座プロファイル | AI根拠提示・最終承認 |
| **D 税理士↔事務所後任** | 既存設計の副次効果 | 対象外（専門知識は範囲外） | 過去仕訳照合DB・運用ノウハウ記録（核） | 既存設計の適用（新規なし） | 既存設計の副次効果 |

20セルすべてに「対応する機能」または「対象外と判断した理由」が存在し、漏れなく埋まっている。

## 8. 本システムが扱う範囲（最終結論）

**扱う**：税額に影響する、または属人性・引き継ぎロスにつながる非対称
- 4区分確認、過去仕訳照合、AI仕訳変換、証拠書類収集、チェックリスト
- 資料不足検知、口座・カードプロファイル、税理士向け確認・承認フロー
- 未連携口座管理、入手方法記録

**扱わない**：
- 会計知識を顧問先に求める方向の非対称解消（教育的アプローチ）
- 経営助言・予測（税額予測、異常検知、制度改正通知）
- 事務所の内部業務管理（顧問先との非対称とは無関係）
- 感覚的判断で十分代替できる定量化（協力度合い等）

## §4. システム全体構成 [全体]

```
[Webアプリ（Vue）]
　├─ 領収書・通帳明細アップロード（顧問先操作）
　├─ OCR処理（Claude/Gemini）
　├─ MCP経由でMF（マネーフォワード クラウド会計）から仕訳データを同期
　├─ 独自システムDB（同期した仕訳データを保存）
　├─ 過去仕訳照合エンジン（DB内データとの完全一致検索）
　├─ 機械的エッジケース判定エンジン（Yes/No条件による判定）
　├─ 顧問先向け確認フォーム（入金4択/出金5択＋資料添付＋メモ）
　├─ AI仕訳変換エンジン（Claude/Geminiによる事実→勘定科目の変換）
　├─ 税理士向け確認画面（要確認フラグ付き）
　└─ MFへの仕訳登録（MCP経由）
```

### データ同期の構成 [C-③]

| トリガー | 内容 |
|---------|------|
| 自動（定期実行） | MCP経由でMFの最新仕訳データを独自システムDBに同期 |
| イベント駆動（顧問先の保存操作時） | 顧問先がフォームで確定させた瞬間、MCP経由でMFへ書き込み、独自システムDBも即時更新 |

> [!WARNING]
> **残課題 R-3**: 両トリガーが同一取引に対しほぼ同時に発生した場合の正データ判定ルール（タイムスタンプ優先かMF優先か）は未確定。
> 現在のreceipt-appで既にDL-050事故（更新経路2本問題）を起こしている。同じ構造を再現しないよう、**更新経路を1本に確定させる設計が必須**。

## §5. 仕訳処理フロー（3層構造）[A-①②, C-②③]

### 全体フロー（最終確定版）

```
領収書/クレカ/通帳明細 取り込み
　↓
①過去仕訳照合 → 特定できた：自動確定（聞かない）
　↓ できない
②機械判定：入金/出金（符号・データ源から自動判定、聞かない）
　↓
③税区分混在チェック（軽減税率・非課税・不課税）→ 混在あり：追加資料が必須
　↓
④顧問先確認フォーム（出金5択／入金4択、アコーディオン形式、1回で完結）
　↓
⑤AI仕訳変換（経費・売上・その他収入のみAI推論／仕入・資産・貸し借り系は固定マッピング）
　↓
⑥税理士が確認・承認
　↓
⑦MFへ登録、独自システムDBに記録（次回から①でヒット）
```

## §6. ①過去仕訳照合（旧称「学習」の正しい定義）[C-③, D-③]

> **定義：初取引時に確定した正規化済み取引先名と勘定科目の組み合わせをDBに記録し、以降は新規取引の取引先名をそのDBと照合（完全一致検索）することで、過去仕訳に基づく自動仕訳を行う仕組み**

- AIモデルのファインチューニング・継続学習は含まない（単純なルールベースの検索）
- 「学習」という語は誤解を招くため設計ドキュメントから排除し、「**過去仕訳照合（DB蓄積型）**」と表現する

判定条件（すべてYesなら自動確定）：
- 取引先名が過去仕訳と一致
- その取引先に対する勘定科目が1パターンしかない

### 用語の正確な使い分け

| 場面 | 正しい用語 |
|------|-----------| 
| ①過去一致判定の仕組みそのもの | 過去仕訳照合 |
| 確定した取引をDBに記録する行為 | 仕訳パターンの蓄積 |
| 結果として確認件数が減る現象 | 「照合対象が増えるため、確認が必要な件数が減少する」 |

> [!WARNING]
> **残課題 R-4**: 「1パターンしかない場合のみ自動確定」は、自動確定率が極端に低くなる可能性がある。
> 現実にはほとんどの取引先で複数科目が使われる（例：ヤマダ電機→消耗品費/備品/修繕費）。
> エッジケース#2（同一店名で科目が複数）が実はエッジケースではなく主流パターンである可能性が高い。

## §7. ②機械的エッジケース判定 [A-①]

確信度（曖昧な中間層）は使わず、すべて機械的なYes/No条件で判定する。

| # | エッジケース | 判定方法 | 証拠資料の必要性 |
|---|------------|---------|----------------|
| 1 | 人名のみの取引 | 取引先名に法人格・既知店舗パターンがない | 推奨（外注/立替の証拠） |
| 2 | 同一店名で科目が複数 | 過去仕訳で同一取引先に複数科目の履歴がある | 不要（選択式で解決） |
| 3 | 決済代行業者 | 既知リスト（PayPal、Square等）と一致 | 必須（真の取引相手の証拠） |
| 4 | 判読不能な略称 | カナ略称・コードのみで意味を持たない文字列 | 推奨 |
| 5 | 海外取引 | 取引先国コードが海外 | 必須（インボイス・消費税区分判定） |
| 6 | 金額帯（資産計上判定） | 金額が一定閾値（10万円等）を超える | 推奨（一括/分割償却の判定） |
| 7 | 法人名だが個人実態 | 機械判定不可（「その他」蓄積から定期見直し） | 必須（契約書等） |

> [!NOTE]
> エッジケース#1（人名判定）と#4（判読不能な略称）は、実質的にAI/NLPタスクであり「機械的Yes/No」との宣言に矛盾する面がある（残課題 R-8）。

## §8. 顧問先確認フォーム（最終確定形）[A-①②]

### 設計の変遷

| 版 | 内容 | 問題 |
|---|---|---|
| 第1版 | 5択（仕入/経費/立替A/立替B/資金移動） | 「立替」が会計用語、資金移動が分かりにくい |
| 第2版 | 立替を事実ベース表現に言い換え | 役員借入/貸付の「返済・回収（清算）」が拾えない |
| 第3版 | 「新規発生」と「清算」を分離、入金/出金で選択肢セット切替 | 全8パターン完全網羅 |
| **最終版** | アコーディオン形式、説明文＋具体例付き、最大1回で完結 | **現行** |

### 4つの会計概念の定義（内部・税理士向け）

| 概念 | 意味 | 主体 |
|---|---|---|
| 役員借入金 | 会社が役員からお金を借りている（会社の負債） | 法人 |
| 役員貸付金 | 会社が役員にお金を貸している（会社の資産） | 法人 |
| 事業主借 | 個人事業主が、事業のお金として自分のお金を入れた | 個人事業主 |
| 事業主貸 | 個人事業主が、事業のお金を個人的に使った | 個人事業主 |

### 入金/出金の自動判定 [B-①]

顧問先に「これは入金ですか出金ですか」と聞く必要は一切ない。

| データソース | 判定方法 |
|---|---|
| 領収書 | 出金のみ（入金の領収書は存在しない） |
| クレカ明細 | 利用＝出金、返金・キャッシュバック＝入金（符号で判別） |
| 通帳明細 | 入金/出金が明細データに符号（＋／－）として明示 |

### 確定：顧問先向け選択肢

#### 出金側（5択）

```
[1] 仕入
　売るための商品・材料を買った
　例：店舗で売る商品、製造に使う材料

[2] 経費
　事業を運営するために使った
　例：消耗品、交通費、会食、事務用品

[3] 自分（役員・事業主）が会社・事業のお金を使った
　例：役員貸付、事業主貸

[4] 会社・事業が借りていたお金を返した
　例：役員借入金の返済

[5] 資産
　長く使うもの・大きな買い物
　例：パソコン、車、机・椅子（10万円以上）
```

#### 入金側（4択）

```
[1] 売上
　商品・サービスを提供して受け取った
　例：販売代金、サービス利用料

[2] 自分（役員・事業主）が会社・事業にお金を入れた
　例：役員借入、事業主借

[3] 会社・事業が貸していたお金を返してもらった
　例：役員貸付金の回収

[4] その他の収入
　本業以外の収入
　例：補助金、保険金、利息
```

### 全8パターンの対応確認（漏れチェック）

| 概念 | 対応する選択肢 |
|---|---|
| 役員借入金（発生） | 入金[2] |
| 役員借入金（返済） | 出金[4] |
| 役員貸付金（発生） | 出金[3] |
| 役員貸付金（回収） | 入金[3] |
| 事業主借 | 入金[2] |
| 事業主借の解消 | 出金[4] |
| 事業主貸 | 出金[3] |
| 事業主貸の解消 | 入金[3] |

**全8パターン、漏れなし。**

│ ○ 自分が会社・事業のお金を使った  ▽   │
│ ○ 会社・事業が借りていたお金を返した ▽ │

│                                 │
├─────────────────────────────┤

### 設計原則

- 選択肢名に会計用語（役員借入、事業主貸等）は出さず、例として括弧内に小さく示すだけ
- 顧問先は「自分が会社にお金を入れた」という生活実感だけで選べる
- 入金/出金の判定は機械的に行い、顧問先には聞かない
- 過去仕訳照合で「特定できた」場合は、この選択肢自体を顧問先に見せない
- アコーディオン形式：選択肢名は常に表示、説明文・例はタップで展開

> [!IMPORTANT]
> **残課題 R-1（🔴致命的）**: 出金[1]仕入と[2]経費の区別は、顧問先に会計判断を求めている。
> 「仕入＝売上に直結する商品・材料の購入（原価）」は事実ではなく会計判断。
> → 仕入/経費を統合して「事業のために使った」の1択にし、仕入/経費の判断はAI+税理士に任せる案を検討すべき。

> [!IMPORTANT]
> **残課題 R-2（🔴致命的）**: 出金[5]資産は、金額閾値（10万円超）で機械判定し選択肢から外す案がある。
> 顧問先に「長く使うものかどうか」を判断させるのは、実質的に会計知識を求めている。

## §9. 証拠書類の設計 [A-④]

### 大原則

**「添付されなくても処理を止めない」**

- 証拠書類は常に「任意」として提示し、添付がなくても次の処理に進む
- 添付があればAIが読み取り精度向上、なければ仮置きルールで処理し「要確認」フラグを立てる
- 100%添付されることには期待しない。添付されたらラッキー

### 証拠書類が必要になる場面：2軸の分離

#### 軸A：初回取引（契約型）— 1回確認すれば以後不要

| # | 場面 | 必要な理由 |
|---|---|---|
| 1 | 融資・借入 | 元本/金利の分解が金額だけでは不可能。契約書で返済スケジュール確定 |
| 2 | 家賃・リース | 事業用割合・礙金の按分が必要。契約書で按分ルール確定 |
| 3 | 減価償却（固定資産購入） | 耐用年数・償却方法の決定に仕様書等が必要 |

#### 軸B：内訳がないと取引内容が分からない — 取引ごとに必要

| # | 場面 | 必要な理由 |
|---|---|---|
| 4 | 入出金額の中に手数料などが混じっている | 手数料の分解が金額だけでは不可能 |
| 5 | 食材などの軽減税率が混じっている | 8%/10%の内訳が金額だけでは不可能 |
| 6 | 輸出 | 消費税区分（課税/免税/対象外）の判定に必要 |

### 仮置きルール（添付なしの場合）

顧問先に明示的に宣言する仮置きルール：

| 条件 | 仮置き |
|---|---|
| 消費税率 | **10%で統一** |
| 入金額 | **手数料等の差引なし**（記載金額がそのまま売上・支払額） |

その他の仮置き（内部処理、顧問先には見せない）：

| ケース | 仮置き |
|---|---|
| 融資・借入 | 全額をいったん「借入金（仮）」で処理、金利分解は税理士が確認 |
| 家賃 | 全額をいったん「事業用100%」で処理、按分が必要なら税理士が修正 |

### 顧問先向け添付欄の文言（最終版）

```
📎 証拠書類を添付（あれば）

次のような場合は、資料の添付をお願いします：

【初回取引】
・融資を受けた、家賃の支払い、減価償却

【内訳がないと取引内容が分からない場合】
・入出金額の中に手数料などが混じっている
・食材などの軽減税率が混じっている
・輸出

※資料の添付がない場合は消費税10%で統一、
　手数料等は差し引かれていない前提で処理します
```

### 添付欄の表示/非表示制御

- **過去に共有済みの取引**: 添付欄自体を表示しない
- **初めての取引**: 添付欄を表示（上記の文言付き）

### 証拠書類確認履歴DB

```
独自システムDB
　└ 証拠書類確認履歴テーブル
　　　取引先名／契約ID／確認日／確認内容（按分比率・元本金利スケジュール等）
　　　↓
新しい取引が同じ取引先・契約から来た場合
　　　→ 確認履歴を検索し、あれば証拠書類を再要求せず、過去の確認内容を適用
```

## §16-3. 口座・カードプロファイル [C-④, D-③]

### 問題

税理士は「何が確認待ちか」を都度見ないと分からない。担当交代時に「この顧問先はこの口座をこう資料でもらう」という運用ノウハウが失われる。

### 機能

| 項目 | 内容 |
|---|---|
| 主な取引先 | 過去仕訳から自動集計 |
| 入金/支払サイト | 過去の入出金日パターンから推定 |
| 定期支払の有無 | 月次で繰り返し発生する取引の検出 |
| 受領方法の記録 | 未連携口座テーブルの `receivingMethod` と連動 |
| 更新トリガー | 月末（定期）＋ページを開いたタイミング（イベント駆動） |

### D-③への寄与

担当が変わっても、プロファイルを見れば「この顧問先はこの口座をCSVで毎月もらう」「定期支払はこのパターン」が即座に分かり、属人性を排除する。
