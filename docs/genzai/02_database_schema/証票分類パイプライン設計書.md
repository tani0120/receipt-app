## 3. 前処理パイプライン（preprocess.ts）

### 3.1 ファイル種別の判定

| 拡張子 | カテゴリ | 処理ルート |
|--------|---------|-----------|
| .jpg .jpeg .png .bmp .tiff .webp .heic | `image` | sharp pipeline |
| .pdf | `pdf` | 直送（Geminiネイティブ対応） |
| .csv | `csv` | Shift-JIS/UTF-8変換 |

### 3.2 sharp pipeline（画像前処理）

6段階のパイプライン。全ステップが意味を持つ:

| # | メソッド | 目的 | パラメータ |
|---|---------|------|-----------|
| 1 | `.rotate()` | EXIF回転補正 | パラメータなし（EXIF自動判定） |
| 2 | `.resize(2000, null, { withoutEnlargement: true })` | 横幅上限 | 2000px上限。元画像が小さければ拡大しない |
| 3 | `.grayscale()` | 白黒化 | なし。色情報を除去しOCR精度を向上させる |
| 4 | `.normalize()` | コントラスト正規化 | なし。ヒストグラム均一化 |
| 5 | `.sharpen({ sigma: 1.0 })` | 文字エッジ強調 | sigma=1.0。文字の輪郭を際立たせる |
| 6 | `.jpeg({ quality: 85 })` | JPEG出力 | quality=85。ファイルサイズ削減 |

### 3.3 A/Bテスト制御

```typescript
const ENABLE_PREPROCESS = process.argv.includes('--preprocess');
```

- `enablePreprocess=false`: 生画像をそのままBase64エンコードして直送
- `enablePreprocess=true`: sharp pipeline全6ステップを適用

PDF は前処理フラグに関わらず常に直送。Gemini がネイティブでPDFを読めるため。

### 3.4 CSV処理

1. BOMチェック: `EF BB BF` があればUTF-8として処理
2. 高バイト検出: `0x80`以上のバイトがあればShift-JISと仮定
3. `iconv-lite` で `Shift_JIS` → UTF-8にデコード
4. ヘッダー `--- CSV DATA START ---` / フッター `--- CSV DATA END ---` で囲む
5. テキストPartとしてVertex AIに送信

### 3.5 出力

```typescript
interface ProcessedFile {
    fileName: string;          // ファイル名
    category: FileCategory;    // 'image' | 'pdf' | 'csv' | 'unknown'
    part: Part;                // Vertex AI送信用Part（inlineData or text）
    originalSize: number;      // 元ファイルサイズ（bytes）
    processedSize: number;     // 処理後サイズ（bytes）
    preprocessed: boolean;     // 前処理を実行したか
}
```

## 4. Geminiへの指示（SYSTEM_INSTRUCTION + REQUEST_PROMPT）

### 4.1 SYSTEM_INSTRUCTION（ロール定義 + ルール）

**ロール**: 「日本の法人会計に精通したAI公認会計士」兼「高度画像解析エンジン」

**基本原則5つ**:

| # | 原則 | 詳細 |
|---|------|------|
| 1 | 勘定科目制限 | 提供リストの科目のみ使用。リスト外は一切不可 |
| 2 | 3状態の区別 | 値あり→値を入れる / 欄なし→null / 読めない→unreadable=true+null |
| 3 | 数値優先 | 画像上の印字数値を最優先。計算値との不整合があっても画像の数値を採用 |
| 4 | T番号 | 「T」+ 数字13桁。ハイフン除去（例: T6120002074677） |
| 5 | 構造化出力厳守 | 指定JSONスキーマ通りに出力。テキスト解説は不要 |

**証票タイプ判定基準**:

| タイプ | 判定基準 |
|--------|----------|
| RECEIPT | 「領収証」「レシート」の表記、POS端末出力 |
| INVOICE | 「請求書」「御請求書」「納品書兼請求書」 |
| TRANSPORT | 「乗車券」「ICカード利用明細」「タクシー領収書」 |
| CREDIT_CARD | 「クレジットカード利用明細」「カードご利用」 |
| BANK_STATEMENT | 通帳ページ、「普通預金」「当座預金」 |
| MEDICAL | 「診療」「薬局」「医療費」「クリニック」 |
| NOT_APPLICABLE | 名刺、メモ、風景写真、謄本等の仕訳不要書類 |

**銀行名推定の手がかり**:
- ロゴの形状・配色・フォント
- 列ヘッダーの固有表現（「お支払金額」「お預り金額」等）
- 摘要欄の略称・固有キーワード
- 通帳の地紋パターン
- 推定理由を `bank_name_evidence` に記載

**仕訳候補のルール**:
- `journal_entry_suggestions` は配列
- `entry_type`: `"debit"`（借方）または `"credit"`（貸方）
- 通常: debit 1件 + credit 1件 = 合計2件
- 複合仕訳（`is_composite_transaction=true`）: 3件以上
- 借方合計と貸方合計は必ず一致
- NOT_APPLICABLEは空配列

**明細の使い分け**:
- レシート → `receipt_items`（全品目）
- 通帳/カード明細 → `line_items`（各取引行）
- それ以外 → 両方null

### 4.2 REQUEST_PROMPT（実行指示）

**会計コンテキスト**:
- 会計期間: 2025-04-01 〜 2026-03-31
- 勘定科目リスト（27科目、これ以外は使用禁止）:

```
旅費交通費, 消耗品費, 通信費, 会議費, 接待交際費, 広告宣伝費, 支払手数料,
地代家賃, 水道光熱費, 保険料, 修繕費, 雑費, 福利厚生費, 外注費, 荷造運賃,
売上高, 仕入高, 現金, 普通預金, 売掛金, 買掛金, 未払金,
仮受消費税, 仮払消費税, 医療費, 受取利息, 不動産収入
```

**実行手順（11ステップ）**:

| # | ステップ | 内容 |
|---|---------|------|
| 1 | 証票分類 | 7タイプから1つ選択。信頼度0.0-1.0 |
| 2 | OCR抽出 | 日付・金額・発行者を抽出。読めなければunreadable=true |
| 3 | 支店名 | 「○○店」「○○支店」があればissuer_branchに |
| 4 | 支払方法 | 現金/クレジット/振込/電子マネー/QR決済を判定 |
| 5 | 適格請求書 | T番号検索。ハイフン除去 |
| 6 | 税率抽出 | 8%/10%の内訳をtax_entriesに。複数税率は複数エントリ |
| 7 | 銀行推定 | 通帳の場合、ロゴ・フォーマットから推定。根拠をevidenceに |
| 8 | 仕訳生成 | 借方・貸方を配列で。複合はis_composite_transaction=true |
| 9 | 明細展開 | 通帳/カードはline_items、レシートはreceipt_items |
| 10 | 手書き判定 | handwritten_flagで3値判定。MEANINGFUL=会計的に意味のある手書き、NON_MEANINGFUL=チェック等、NONE=なし。角印・スタンプ除外 |
| 11 | 複数証票 | 1画像に2枚以上ならhas_multiple_vouchers=true |

### 5.2 サブ型定義

**TaxEntry**（税率明細）:
```typescript
{ rate: number; taxable_amount: number; tax_amount: number; }
```

**JournalSuggestionEntry**（仕訳候補1件）:
```typescript
{ entry_type: 'debit' | 'credit'; account: string; amount: number; description: string; }
```

**ReceiptItem**（レシート商品1件）:
```typescript
{ name: string; quantity: number|null; unit_price: number|null; amount: number; tax_rate: number|null; }
```

**LineItem**（通帳/カード明細1行）:
```typescript
{ date: string|null; description: string; amount: number; debit_account: string|null; credit_account: string|null; }
```

### 5.3 Vertex AI JSON Schema（CLASSIFY_RESPONSE_SCHEMA）

`@google-cloud/vertexai` の `SchemaType` を用いた構造化出力スキーマ。Geminiのレスポンスを強制的にこの構造に合わせる。

- 全29フィールドを `properties` で定義
- `nullable: true` でnullを許容するフィールドを明示
- `required` 配列でGeminiに必ず出力させるフィールドを指定
- enum型はstring + enum配列で制約

## 6. 後処理（classify_postprocess.ts）

### 6.1 層Bの設計思想

「Geminiにやらせるな、コードでやれ」が原則。数学的に100%正確に判定できる項目はGeminiに頼まない。

### 6.2 各項目の詳細

#### B1: classification_status（ステータス判定）

判定ロジック:
```
1. is_not_applicable=true → "excluded"
2. voucher_type_confidence < 0.6 → "needs_review"
3. いずれかのunreadable=true → "needs_review"
4. tax_calculation_mismatch=true → "needs_review"
5. debit_credit_mismatch=true → "needs_review"
6. 上記に該当しない → "auto_confirmed"
```

#### B2-B3: tax_calculation_mismatch / tax_calculation_detail（税検算）

```
各tax_entryについて:
  expected_tax = taxable_amount × rate ÷ 100（端数切り捨て）
  合格条件: |expected_tax - tax_amount| ≤ 1（端数許容）

合計検算:
  sum(taxable_amount + tax_amount) == total_amount
  合格条件: |差| ≤ entries.length（端数累積許容）
```

出力例: `"10%: 9091+909=10000 → 合計10000=10000 ✅"`

#### B4-B5: date_anomaly / date_anomaly_reason（日付異常）

```
会計期間: 2025-04-01 〜 2026-03-31
条件: date が期間外 → anomaly=true
理由例: "2025-03-15 は期首2025-04-01より前"
```

#### B6: duplicate_suspect（バッチ内重複検出）

```
全件のGemini結果を受け取り、同一バッチ内で:
  date + total_amount + issuer_name が一致する件を検出
一致 → duplicate_suspect=true
```

バッチ外（過去処理済み）との重複検出は層C（DB必要）のスコープ。

#### B7-B8: debit_credit_mismatch / debit_credit_detail（貸借検算）

```
sum(debit entries) == sum(credit entries)
不一致 → mismatch=true
出力例: "借方10000=貸方10000 ✅"
```

#### 料金推定: estimated_cost_usd

```
Gemini 2.5 Flash の料金体系に基づくトークン単価計算:
  入力: $0.15 / 1M tokens（128Kまで）
  出力: $0.60 / 1M tokens
  思考: $3.50 / 1M tokens

推定料金 = (prompt × 0.15 + completion × 0.60 + thinking × 3.50) / 1_000_000
```

## 9. 層の分離原則

| 層 | 誰が処理 | 入力 | 出力 | 正確性 |
|----|---------|------|------|--------|
| A（29フィールド） | Gemini | 画像/PDF/CSV | JSON | 確率的（confidence付き） |
| B（8項目） | コード | 層Aの出力 | JSON | 100%数学的に正確 |
| C（スコープ外） | DB + コード | 層A+B + 過去データ | — | DB依存 |

**層Bをコードにした理由**:
- `classification_status`: 閾値判定は数学。Geminiに聞く意味がない
- `tax_calculation_mismatch`: 足し算の検算。Geminiに計算させると間違える
- `date_anomaly`: 日付比較。コードの方が100%正確
- `duplicate_suspect`: バッチ全件の突合。Geminiは1ファイルしか見えない
- `debit_credit_mismatch`: 合計比較。算数はコードでやる

**層Cが今スコープ外の理由**:
- 過去仕訳との重複検出 → DB必要
- 過去平均との比較 → DB必要
- 証票欠落検出 → DB必要

## 10. 技術的制約と既知の問題

| # | 項目 | 詳細 | 対応 |
|---|------|------|------|
| 1 | thoughtsTokenCount | Vertex AI SDKの型定義に含まれていない | `usage as Record<string, unknown>` でキャスト（H1） |
| 2 | tsconfig不整合 | src/scripts/ がDOM用tsconfigの下にある | tsconfig.pipeline.json で分離予定（P1-P3） |
| 3 | Set型エラー | tsx実行時にSet初期化で型推論失敗 | 配列リテラル + includes() に書き換え済み |
| 4 | sharp CJS/ESM | `import sharp from 'sharp'` がesModuleInterop依存 | NodeNext設定で解消見込み |
