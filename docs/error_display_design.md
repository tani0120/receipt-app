# ロール別エラー表示設計

> 作成日: 2026-04-23
> 前提: スタッフ・顧問先専用システム。一般公開なし。

---

## 1. 設計原則

- **第三者**: 存在を隠す。情報ゼロ。次の行動も示さない
- **顧問先**: 迷子を助ける。次の行動だけ示す。理由は出さない
- **スタッフ**: 業務継続のために全開示。対応手順まで示す
- **IPアドレスは使わない。JWTで完結**
- **403より404。存在自体を隠す**

---

## 2. ロール定義

| ロール | 状態 | 判定方法（暫定） | 判定方法（Supabase移行後） |
|---|---|---|---|
| 第三者 | 未ログイン | `auth.getUser()` が null | 同左 |
| 顧問先 | ログイン済み | `guestAllowed` メタ + `guest_google_*` | `app_metadata.role === 'client'` |
| スタッフ | ログイン済み | JWT有り + `guestAllowed`でないルート | `app_metadata.role === 'staff'` |

## 4. 全ロール共通の禁止事項

| 禁止事項 | 理由 |
|---|---|
| スタックトレース（第三者・顧問先向け） | 内部構造の露出 |
| フレームワーク名・バージョン | 攻撃の足がかり |
| DBエラー文字列 | テーブル名・カラム名の漏洩 |
| 「スタッフ専用です」（第三者向け） | 存在を示唆する |
| 「ログインが必要です」（第三者向け） | 同上 |

## 6. SPA固有の注意事項

このシステムはVue SPAのため、ルーター遷移ではHTTPステータスコードを返せない（常に200）。
APIレスポンスのステータスコードでフロント側がエラーコンポーネントを出し分ける設計とする。

```
API → { error: "安全な日本語文面", requestId: "abc-123" }（apiError/apiCatchError経由）
  ↓
apiFetch → normalizeHttpError → AppError { コード, メッセージ, パス, リクエストID }
  ↓
400系 → ページ遷移しない。呼び出し元でUI表示（toast/modal）
401   → /login に自動遷移
404/500系 → /404 に自動遷移 → ロール判定 → 該当するエラーコンポーネント表示
```
