## ⚠️ 重要：このドキュメントの位置づけ

**これは何か**:
- 「人間の迷い・感覚・曖昧さ」を**構造だけ保った表**
- プロパティ名・型・実装を一切書かない
- 判断基準を数値化しない

**これは何ではないか**:
- ❌ プロパティの意味定義
- ❌ UI仕様書

## なぜ「表」形式なのか？

### ❌ 文章だけはダメ

- 後でAIが構造を誤解する
- 意味定義に引きずられる
- 比較できない

### ⭕ 表が強い理由

- **比較できる**: 複数UseCase間で欠けている項目が分かる
- **視覚的**: 欠けている項目が一目で分かる
- **射影可能**: 後でproperty-integration-map.mdに対応付けられる

## 🎯 Phase 3-A: UseCaseを作った目的

### ❌ 目的ではないもの

- プロパティの説明書を作ること
- 仕様を確定させること
- 正解の業務フローを決めること

### ✅ 本当の目的（1行）

```
UIモックで"どこに判断の重心があるか"を暴くため
```

**もう少し正確に言うと**:

人間が
- 立ち止まる
- 迷う
- 違和感を覚える

その瞬間に
👉 **どんな情報が、どんな形で無いと事故るか**

これを **人間の言葉で先に固定するため**。

**UseCaseは**:
- ❌ 「正解を決めるため」ではなく
- ✅ 「**壊れやすい箇所を特定するため**」に作っています

## 📊 Phase 3-A-2: UseCase 3レイヤー化

### 判定の単位：Decision Cluster

**判定の単位は「UseCase単体」ではなく**  
👉 **「判断行為のグループ（Decision Cluster）」**

### 判定ルール

#### Step 1：全207件を「判断の種類」で束ねる

- 仕訳処理を判断する系
- 作業可否・進行可否を判断する系
- 契約・顧問関係を確認する系
- 数値・日付・状態を参照するだけ系

**※ この時点では core / reference を決めない**

**🔒 運用ルール①：グループ粒度は4つのみ**
- 細分化しない（思考のブレーキ）
- Step 1 = 思考停止して投げ込む箱

---

#### Step 2：そのグループに「独立した判断UI」が必要か？を問う

```
❓ この判断は「人が立ち止まって考えるための画面」を個別に設計する価値があるか？
```

- **YES** → このグループは **core**
- **NO** → このグループは **reference or log**

**🔒 運用ルール②：グループごとに承認**
- 一括承認はNG
- 承認の単位 = Decision Cluster（UseCase単体ではない）

---

#### Step 3：グループ内で役割分解

- **判断の主役** → core
- **判断材料として表示されるだけ** → reference
- **履歴・裏取り・証跡** → log

**🔒 運用ルール③：UseCaseは修正しない**
- **全UseCaseはRaw Dataとして保持**
- UseCaseの文言は修正しない（違和感という証跡を保持）
- **Mapping TableでIDに役割を振るだけ**

---

### � 最新版フロー（UIモック作成まで）

**Step 1**: 全207件を4箱に投げ込む ✅ 完了

**Step 1.5**: 4箱をサブクラスタに分ける（認知状態で切る）
- 立ち止まる判断 / 流れ内確認 / 記録

**Step 2**: サブクラスタ単位で「判断主役UIか？」を問う（修正版）
```
❓ このサブクラスタは「人が決める瞬間を支えるための判断主役UI」として
   独立した画面を設計する価値があるか？
```
- YES → core
- NO → reference/log

**Step 3**: coreサブクラスタのみ内部役割分解
- 判断の主役 → core
- 判断材料 → reference
- 履歴・証跡 → log

**成果物**: Mapping Table（UseCaseは修正しない）

**期待される結果**: coreが30〜40件に収束 → Phase 3-B（UIモック）へ

## Use Case: UC-002 顧問先のステータスから作業可否を判断したい

| 項目 | 内容 |
|---|---|
| **利用者** | マネージャー／担当者 |
| **タイミング** | 契約解除後、契約前、作業開始前 |
| **目的** | この顧問先で作業を進めてよいか判断したい |
| **きっかけ** | 契約が解除された／契約前だが作業依頼がきた |
| **行動** | 顧問先一覧でステータスを確認する |
| **判断** | 作業を進めるか、保留するか、上長に相談するか |
| **許容曖昧さ** | ステータスだけで判断（詳細な契約内容は不要） |
| **間違うと困ること** | 契約外の作業をして請求できない、または契約中なのに作業を止めてクレームになる |
| **代替手段** | 電話・メール・チャットワーク確認、記憶、上長に都度確認 |

**補足**:
- システムはアクティブじゃない顧問先の作業を実施しない
- 顧問先一覧ページではステータスによって表示を切り替える

## Use Case: UC-003 契約開始日を確認したい（判断はしない）

| 項目 | 内容 |
|---|---|
| **利用者** | 担当者／マネージャー |
| **タイミング** | 顧問先一覧を眺めるとき |
| **目的** | ただ確認するために見る（判断のためではない） |
| **きっかけ** | 顧問先一覧を開いた |
| **行動** | 契約開始日の列を見る、ソートする |
| **判断** | 判断はしない（ただ確認するだけ） |
| **許容曖昧さ** | 正確な日付でよいが、自動判断には使わない |
| **間違うと困ること** | 特になし（参照用のため） |
| **代替手段** | 契約書を見る、記憶 |

**補足（理想と現実のブレ）**:
- **理想定義**: 契約開始日 = ロジックの起点
- **現実のブレ**: 誰もその日付で自動判断してない。ただ「目で見て確認」してるだけ

**システムの挙動**:
- 顧問先一覧をソートする機能のみ

## Use Case: UC-004 決算月を起点に業務を管理したい

| 項目 | 内容 |
|---|---|
| **利用者** | 担当者／マネージャー／システム |
| **タイミング** | 仕訳処理時、顧問先一覧確認時、業務管理時 |
| **目的** | 期をまたぐ仕訳処理の判断、繁忙期把握、人員配分 |
| **きっかけ** | 期首期末の仕訳を処理する、決算月ごとの集計を見る |
| **行動** | 決算月を確認、顧問先一覧を決算月でソート、決算月別の顧問先数を集計 |
| **判断** | 期首期末の売掛買掛等をどう処理するか、来月の繁忙期に備えて人員配分するか |
| **許容曖昧さ** | 正確な決算月が必要（システム判断に使うため） |
| **間違うと困ること** | 期ズレで税務処理ミス、繁忙期に人員不足 |
| **代替手段** | 契約書確認、決算月を手動で集計 |

**補足（理想と現実のブレ）**:
- **理想定義**: 決算月 = 税務処理用
- **現実のブレ**: 業務管理、人員配分、繁忙期把握にも使っている

**システムの挙動**:
- 仕訳の際に期首期末の売掛買掛等の判断に使う
- 顧問先一覧を決算月で昇順に並べる
- 決算月ごとの顧問先数、担当者別の決算月別の顧問先数を集計

## Use Case: UC-005 顧問先を一意に識別したい（システム内部）

| 項目 | 内容 |
|---|---|
| **利用者** | システム（自動処理） |
| **タイミング** | 顧問先登録時 |
| **目的** | 顧問先を他の顧問先と区別する一意のIDを付与 |
| **きっかけ** | 顧問先が新規登録された |
| **行動** | システムが自動採番してIDを付与 |
| **判断** | 判断はしない（自動処理） |
| **許容曖昧さ** | 一意性が保証されれば形式は問わない |
| **間違うと困ること** | IDが重複すると顧問先の仕訳や報酬が混在する |
| **代替手段** | なし（システム必須） |

**補足**:
- 顧問先ごとの報酬や作業時間、コストなどを把握するために使用
- **顧問先IDは画面に表示不要**（内部利用のみ）
- 3コードとは別の概念

## Use Case: UC-006 守秘義務を守りながら顧問先を特定したい（3コード）

| 項目 | 内容 |
|---|---|
| **利用者** | 担当者／マネージャー |
| **タイミング** | 顧問先登録時、社外での会話時、請求時 |
| **目的** | 顧問先の話を社外でするときに会社名を使わず、守秘義務を守る |
| **きっかけ** | 社外で顧問先について話す必要がある、システム外で請求する |
| **行動** | 3コードを手入力、閲覧・確認 |
| **判断** | 判断はしない（参照・確認のみ） |
| **許容曖昧さ** | 3コード必須、正確性が必要 |
| **間違うと困ること** | 会社名で話すと守秘義務違反、請求時に顧問先を特定できない |
| **代替手段** | 会社名（守秘義務リスク）、記憶 |

**補足**:
- **3コードは画面に表示が必要**（人間が確認するため）
- システム外の請求時に3コードをIDとして使用
- **3コードは会社名変更等で変わるので、システム内部の一意のIDにしてはダメ**
- 顧問先IDとは別の概念

| **許容曖昧さ** | 処理優先度でOK |
| **間違うと困ること** | 全件即時処理でコスト高、全件バッチで緊急案件の遅延 |
| **代替手段** | イベント駆動のみ（コスト高）、スケジューラーのみ（即時性なし） |

**本システムの主役**: スケジューラーによる定期バッチ（コストメリット）
