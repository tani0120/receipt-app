## ⚠️ 重要：このドキュメントの位置づけ

**これは何か**:
- 「人間の迷い・感覚・曖昧さ」を**構造だけ保った表**
- プロパティ名・型・実装を一切書かない
- 判断基準を数値化しない

**これは何ではないか**:
- ❌ プロパティの意味定義
- ❌ UI仕様書
- ❌ 実装設計書

---

## なぜ「表」形式なのか？

### ❌ 文章だけはダメ

- 後でAIが構造を誤解する
- 意味定義に引きずられる
- 比較できない

### ⭕ 表が強い理由

- **比較できる**: 複数UseCase間で欠けている項目が分かる
- **視覚的**: 欠けている項目が一目で分かる
- **射影可能**: 後でproperty-integration-map.mdに対応付けられる

---

## この表の使われ方

**Phase 3-B（UIモック）**:
- この表を1行ずつUIに変換
- 「この判断、画面でできないな？」を炙り出す

**Phase 3-C（意味定義確定）**:
- 初めて「どのプロパティがどのUseCaseを支えるか」を対応付ける
- property-integration-map.mdの「意味」列を更新

---

## 禁止事項（IDE向け・厳守）

**IDEに対する指示**:
> 「この表は意味を決めるためのものではない。
> プロパティ名を推測して埋めない。
> 判断基準を数値化しない。
> UI構成を確定させない。」

**人間に対する注意**:
- プロパティ名を書かない
- 型を書かない
- 実装・画面名を書かない
- 判断基準を書かない（「感覚」「傾向」で十分）

## Client関連: 業務判断を含みそうな項目

### 報酬・契約関連
**問い**:
- 顧問先のステータス（active/inactive/suspension）は、いつ・誰が・何の判断に使うのか？
- 契約開始日は、請求判断なのか、作業可否判断なのか？
- 決算月は、税務処理のタイミング判断のみか？

### Drive連携・フォルダ管理関連
**問い**:
- Drive連携IDは、証憑管理目的か、監査対応目的か？
- 連携状態は、誰がどのタイミングで確認するのか？

### 税務設定関連（PENDING項目）
**問い**:
- 税務設定は、いつ・誰が・どのタイミングで参照するのか？
- インボイス登録の有無は、仕訳入力時に自動判定に使うのか？

## Use Case: UC-001 顧問先ごとの採算感覚を掴みたい

| 項目 | 内容 |
|---|---|
| **利用者** | マネージャー |
| **タイミング** | 月末 |
| **目的** | この顧問先が儲かっているか直感的に把握したい |
| **きっかけ** | 今月忙しかった感覚がある |
| **行動** | 顧問先一覧をざっと眺める |
| **判断** | 「これは赤字っぽい」「これは問題なさそう」 |
| **許容曖昧さ** | 正確な原価計算は不要、感覚一致で十分 |
| **間違うと困ること** | 赤字顧問先を放置してしまう |
| **代替手段** | スタッフの愚痴・記憶 |

**備考**: この例では、プロパティ名（想定仕訳数、作業時間、報酬）は一切出てきませんが、Phase 3-Cで自然に紐づきます。

## Use Case: UC-002 顧問先のステータスから作業可否を判断したい

| 項目 | 内容 |
|---|---|
| **利用者** | マネージャー／担当者 |
| **タイミング** | 契約解除後、契約前、作業開始前 |
| **目的** | この顧問先で作業を進めてよいか判断したい |
| **きっかけ** | 契約が解除された／契約前だが作業依頼がきた |
| **行動** | 顧問先一覧でステータスを確認する |
| **判断** | 作業を進めるか、保留するか、上長に相談するか |
| **許容曖昧さ** | ステータスだけで判断（詳細な契約内容は不要） |
| **間違うと困ること** | 契約外の作業をして請求できない、または契約中なのに作業を止めてクレームになる |
| **代替手段** | 電話・メール・チャットワーク確認、記憶、上長に都度確認 |

**補足**:
- システムはアクティブじゃない顧問先の作業を実施しない
- 顧問先一覧ページではステータスによって表示を切り替える

## Use Case: UC-003 契約開始日を確認したい（判断はしない）

| 項目 | 内容 |
|---|---|
| **利用者** | 担当者／マネージャー |
| **タイミング** | 顧問先一覧を眺めるとき |
| **目的** | ただ確認するために見る（判断のためではない） |
| **きっかけ** | 顧問先一覧を開いた |
| **行動** | 契約開始日の列を見る、ソートする |
| **判断** | 判断はしない（ただ確認するだけ） |
| **許容曖昧さ** | 正確な日付でよいが、自動判断には使わない |
| **間違うと困ること** | 特になし（参照用のため） |
| **代替手段** | 契約書を見る、記憶 |

**補足（理想と現実のブレ）**:
- **理想定義**: 契約開始日 = ロジックの起点
- **現実のブレ**: 誰もその日付で自動判断してない。ただ「目で見て確認」してるだけ

**システムの挙動**:
- 顧問先一覧をソートする機能のみ

## Use Case: UC-004 決算月を起点に業務を管理したい

| 項目 | 内容 |
|---|---|
| **利用者** | 担当者／マネージャー／システム |
| **タイミング** | 仕訳処理時、顧問先一覧確認時、業務管理時 |
| **目的** | 期をまたぐ仕訳処理の判断、繁忙期把握、人員配分 |
| **きっかけ** | 期首期末の仕訳を処理する、決算月ごとの集計を見る |
| **行動** | 決算月を確認、顧問先一覧を決算月でソート、決算月別の顧問先数を集計 |
| **判断** | 期首期末の売掛買掛等をどう処理するか、来月の繁忙期に備えて人員配分するか |
| **許容曖昧さ** | 正確な決算月が必要（システム判断に使うため） |
| **間違うと困ること** | 期ズレで税務処理ミス、繁忙期に人員不足 |
| **代替手段** | 契約書確認、決算月を手動で集計 |

**補足（理想と現実のブレ）**:
- **理想定義**: 決算月 = 税務処理用
- **現実のブレ**: 業務管理、人員配分、繁忙期把握にも使っている

**システムの挙動**:
- 仕訳の際に期首期末の売掛買掛等の判断に使う
- 顧問先一覧を決算月で昇順に並べる
- 決算月ごとの顧問先数、担当者別の決算月別の顧問先数を集計

## Use Case: UC-005 顧問先を一意に識別したい（システム内部）

| 項目 | 内容 |
|---|---|
| **利用者** | システム（自動処理） |
| **タイミング** | 顧問先登録時 |
| **目的** | 顧問先を他の顧問先と区別する一意のIDを付与 |
| **きっかけ** | 顧問先が新規登録された |
| **行動** | システムが自動採番してIDを付与 |
| **判断** | 判断はしない（自動処理） |
| **許容曖昧さ** | 一意性が保証されれば形式は問わない |
| **間違うと困ること** | IDが重複すると顧問先の仕訳や報酬が混在する |
| **代替手段** | なし（システム必須） |

**補足**:
- 顧問先ごとの報酬や作業時間、コストなどを把握するために使用
- **顧問先IDは画面に表示不要**（内部利用のみ）
- 3コードとは別の概念

## Use Case: UC-006 守秘義務を守りながら顧問先を特定したい（3コード）

| 項目 | 内容 |
|---|---|
| **利用者** | 担当者／マネージャー |
| **タイミング** | 顧問先登録時、社外での会話時、請求時 |
| **目的** | 顧問先の話を社外でするときに会社名を使わず、守秘義務を守る |
| **きっかけ** | 社外で顧問先について話す必要がある、システム外で請求する |
| **行動** | 3コードを手入力、閲覧・確認 |
| **判断** | 判断はしない（参照・確認のみ） |
| **許容曖昧さ** | 3コード必須、正確性が必要 |
| **間違うと困ること** | 会社名で話すと守秘義務違反、請求時に顧問先を特定できない |
| **代替手段** | 会社名（守秘義務リスク）、記憶 |

**補足**:
- **3コードは画面に表示が必要**（人間が確認するため）
- システム外の請求時に3コードをIDとして使用
- **3コードは会社名変更等で変わるので、システム内部の一意のIDにしてはダメ**
- 顧問先IDとは別の概念
